# ADR-0009: Vision 파서의 멀티 기관 정확도는 사전 검증 코퍼스 없이 dogfood로 점진 검증한다

## Status

Accepted

## Date

2026-05-15

## Context

`ImageStatementParser`는 `institution_identifier`를 컨트롤러에서 받을 수 있지만 (`params[:institution_identifier]`, 기본값 `shinhan_card`), `GeminiVisionParserService`로 전달되는 Vision 프롬프트는 **신한카드 이용대금 명세서 양식에 종속**된 단일 텍스트다. 다른 기관(국민·하나·삼성·우리 등)의 명세서가 같은 프롬프트로 들어가면 필드 매핑이 어긋날 가능성이 있다 — [docs/context/current-state.md의 Needs audit](../context/current-state.md) 항목으로 기록되어 있었다.

이 정확도를 검증하기 위해 검토한 대안은 다음과 같다.

1. **사전 검증 코퍼스 구축** — 5~10개 기관의 실제 명세서 스크린샷 fixture(개인정보 마스킹)를 수집해 `test/fixtures/files/`에 두고 정확도 회귀 테스트 작성.
2. **기관별 분기 프롬프트** — `institution_identifier`에 따라 다른 프롬프트를 보내는 dispatcher 추가.
3. **dogfood 점진 검증** — 본인이 실 명세서를 업로드하며 사용하다가 정확도 문제를 만나면 그 시점에 사례별로 프롬프트·파서를 조정한다.

본 프로젝트는 1인 빌드 단계이고, 워크스페이스 사용자도 빌더 본인을 포함한 소수다. 사전 코퍼스 구축은 (1) 개인정보가 들어간 실 명세서를 모으는 절차 자체가 무겁고, (2) 검증해도 측정 가능한 정확도 임계치를 정해두지 않은 시점에서 절대적 의사결정 기준이 없으며, (3) "어떤 기관을 우선할지" 자체가 dogfood로 자연히 정해질 정보다.

## Decision

**Vision 파서의 멀티 기관 정확도는 사전 검증 코퍼스 없이 dogfood로 그때그때 검증한다.** 실 사용 중 잘못 파싱된 명세서를 만나면, 그 케이스를 시작점으로 (1) 프롬프트 수정, (2) `institution_identifier`별 분기 도입, (3) 자동 회귀 fixture 추가 중 어떤 조합으로 대응할지 그 시점에 결정한다.

## Consequences

긍정:
- 도입 비용 0. 실 사용 데이터가 곧 검증 데이터가 된다.
- 정확도 임계치를 인위적으로 미리 정하지 않는다 — 사용자가 "이 정도면 쓸만하다"고 판단하는 실측 기준이 곧 임계치다.
- 개인정보가 포함된 실 명세서를 사전에 수집·저장할 필요가 없다.

부정:
- 빌더 외 사용자가 새 기관 명세서를 처음 올렸을 때 정확도 저하를 먼저 만나게 된다 — 이 케이스는 "실패한 파싱"으로 import 흐름이 막히지 않게 (현재 `FileParsingJob`이 일부 실패도 부분 커밋으로 흡수) 처리하고, 결과 화면에서 사용자에게 명시적으로 노출한다.
- "어느 기관까지 잘 동작하는지" 외부에 광고 가능한 사실로 정리하기 어렵다 — 본 단계에서는 외부 광고 자체를 하지 않으므로 무관.

운영·테스트·문서 영향:
- `docs/context/current-state.md` Needs audit에서 "이미지 파서 멀티 기관 지원" 항목을 본 ADR로 링크하여 마감한다.
- 실 dogfood에서 정확도 문제를 한 번이라도 만나면 그 케이스를 fixture(`test/fixtures/files/`)로 마스킹·저장하고 그 시점부터 회귀 테스트를 누적한다. 일회성 테스트가 곧 코퍼스가 된다.

## 재검토 트리거

다음 중 하나라도 충족되면 본 ADR을 supersede 하는 새 ADR로 사전 검증 코퍼스 도입 여부를 재평가한다.

1. 워크스페이스 활성 사용자가 **빌더 1인 + 가족/팀 소수**를 넘어 외부 사용자가 들어오기 시작한다 (dogfood로 발견할 시간이 부족해진다).
2. dogfood 중 정확도 회귀가 **연 3회 이상** 발생한다 (재현 케이스가 충분해져 사전 코퍼스의 비용이 정당화된다).
3. 기관 분기 프롬프트가 **3개 이상** 분기된다 (분기 자체가 회귀 테스트 없이 안전하게 유지 불가능해진다).

## Alternatives considered

- **사전 검증 코퍼스 구축** — 5~10개 기관 명세서 fixture 수집 후 정확도 회귀 테스트. 거부 이유: 1인 빌드 단계의 도입 비용이 이득을 초과. 실 사용으로 자연 발견될 케이스를 사전에 추측 비용으로 만든다.
- **기관별 분기 프롬프트 선도입** — `institution_identifier` dispatcher로 신한·국민·하나 프롬프트를 미리 만들어두기. 거부 이유: dispatcher 자체는 만들기 쉽지만, **각 기관 프롬프트의 정확도를 보증할 데이터가 없다**. 빈 분기를 미리 만드는 것은 false sense of coverage.
- **외부 OCR 서비스 도입** — Google Document AI, AWS Textract 등. 거부 이유: 비용/지연/외부 의존이 늘고, 현재 Vision 단일 모델로 충분히 동작하는 상태에서 미리 도입할 근거가 없다.
