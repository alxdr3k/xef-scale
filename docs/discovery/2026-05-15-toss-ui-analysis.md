# 토스 앱 UI 해체분석 (Discovery)

- 작성일: 2026-05-15
- 작성자: Claude (analysis session)
- 분류: 외부 UI 디스커버리 / 디자인 리서치
- 권위: **본 레포의 구현 권위가 아님**. 외부 앱(토스/토스증권/토스뱅크) UI를 해체분석하여 추후 디자인 의사결정에 참조하기 위한 노트.

## 목적

토스 앱 9장 스크린샷을 입력으로, UI를 그대로 복제하지 않고 **그 뒤에 깔린 기준·의도·정책·철학·디테일**을 추출한다. 그 다음, 추출한 규칙만으로 원본 스크린의 골조를 재구성 가능한지 시뮬레이션하여 추출 결과의 정합성을 검증한다.

## 입력 스크린

| # | 화면 | 핵심 패턴 |
|---|---|---|
| 1 | 토스증권 홈 | 타이틀 + 인라인 지수 · 캠페인 카드 · 개인화 추천 리스트 |
| 2 | 거래대금 차트 | 스크롤 시 상단 지수 슬롯 교체(VIX) · 광고+랭킹 리스트 · 모드 토글 pill |
| 3 | 관심(빈 상태) | AI 시그널 배너 + 친절한 빈 상태(이름 호명 + 액션 버튼) |
| 4 | 발견 | 카테고리 칩 · AI 이슈 outlined 카드 · 5-탭 세그먼트 |
| 5 | 피드 | 4-탭 세그먼트 · 작성자+뱃지+토픽+팔로우가 한 줄 · 본문 첫 줄 굵게 |
| 6 | 전체 메뉴 | 이름 호명 + 검색 · 바로가기 그리드 · 카테고리 리스트(아이콘+메타) |
| 7 | 설정 | 프로필 헤더 · 그룹 카드(라운드 컨테이너) 구조 |
| 8 | 통장 메인 | 잔액 hero · 이자 인라인 CTA · 최근 거래 미리보기 · 듀얼 CTA |
| 9 | 거래내역 | 잔액 중앙 hero · 풀-와이드 이자 CTA · 날짜 그룹 + 광고 슬롯 · 스티키 듀얼 CTA |
| 10 | 계좌 상세 | 계좌 헤더+복사 · 주요값 인라인 · "내 계좌 관리/이체 관리" 그룹 카드 |

(스크린샷 자체는 본 레포에 저장하지 않음. 분석 시 채팅 첨부로 전달됨.)

## 1. 디자인 시스템 토큰 추출

### 1.1 컬러

| 토큰 | 추정값/역할 | 사용처 |
|---|---|---|
| `bg/base` | 거의 순흑(#0E0F12쯤) | 모든 화면 바탕 |
| `bg/surface` | 한 단계 밝은 회흑(#1B1D22쯤) | 카드, 그룹 컨테이너, 입력창 |
| `bg/elev` | 그보다 살짝 밝음 | pill 토글, 칩, 작은 버튼 |
| `text/primary` | 순백 | 잔액·종목명·타이틀 |
| `text/secondary` | 중간 회색 | 부가설명, 메타 |
| `text/tertiary` | 어두운 회색 | 면책 문구, placeholder |
| `brand/blue` | 토스 시그니처 블루(#3182F6 계열) | 1차 CTA, 링크, 활성 탭, 랭킹 숫자, "하락"(한국식) |
| `semantic/up` | 레드(#F04452 계열) | "상승" — 한국 증권 관행 |
| `semantic/down` | 블루(브랜드 블루 같은 톤) | "하락" |
| `ai/accent` | 라벤더/퍼플 | ✨ AI 마커, AI 카드 outline |
| `product/mint` | 민트그린 | 토스뱅크/용돈/특정 서비스 아이콘 한정 |

**관찰**
- 브랜드 블루 1개로 "1차 액션 / 링크 / 하락 / 랭킹 숫자"까지 다 떠받친다. 다른 의미의 파랑이 별도로 없다 → 신호 과부하 위험을 감수하고 '브랜드 색 일관'을 택함.
- AI는 보라계열로 명확히 분리. 광고는 색이 아니라 라벨("AD", "서비스·광고")로 분리. 즉 **기능적 색**과 **마케팅적 색**을 분리.

### 1.2 타이포

- 한글: Pretendard로 추정되는 기하학적 산세리프. 가변(Variable) 웨이트 활용.
- 위계: **Hero(36~44pt Bold) · Title(24~28 Bold) · Body(16 Regular/Medium) · Caption(13~14)**.
- 숫자 비중이 매우 큼: 잔액/지수/가격이 거의 항상 그 화면의 최대 글자.
- 본문 강조는 **첫 문장만 Bold** 같은 미세 규칙 사용(피드 게시글).

### 1.3 형태/간격

- 코너 라운드: 작은 칩 ~12, 카드 ~16~20, 그룹 컨테이너 ~24, CTA 버튼 풀라운드(pill).
- 카드 패딩: 좌우 ~20, 상하 ~20~24 — 답답함보다 여백을 일관되게 우선.
- 리스트 row: 비교적 큰 64~72pt 높이. 손가락 친화.
- 세이프 에어리어 위에 떠 있는 **원형 백 버튼**과 **고정 탭바**. 백 버튼은 탭바와 같은 baseline.

### 1.4 컴포넌트 카탈로그

| 컴포넌트 | 규칙 |
|---|---|
| Top section header | 좌: 타이틀 + 인라인 라이브 컨텍스트(지수/델타) / 우: 액션 아이콘 트리오(AI ✨ · 검색 🔍 · 메뉴 ☰) |
| Primary CTA | 풀라운드, brand/blue 채움, 흰 글자 |
| Secondary CTA | 풀라운드, bg/surface, 흰 글자 |
| Pill toggle | 어두운 회색 배경 + 흰 글자(예: "간편 홈 보기") |
| Chip(필터) | 작은 라운드 + 활성 시 회색 채움(전체 ⌄) |
| Segmented tab | 텍스트 + 활성 항목 밑줄(거래대금) 또는 chip형(피드) — 컨텍스트별 다름 |
| List row | 좌 아이콘 / 중 타이틀+서브 / 우 값(또는 ❤️) / chevron |
| Ranking row | 좌측에 큰 브랜드 블루 숫자 → 강한 순위 시각화 |
| Group card | 같은 도메인 항목을 하나의 surface 카드로 묶음(설정/계좌 관리) |
| AI card | 보라 outlined 박스 + ✨ 아이콘 + 굵은 한 줄 헤드라인 |
| Promo card | 일러스트(작은 3D 풍) + 타이틀 + 서브 + CTA, 우상단 닫기 X |
| Bottom tab bar | 4~5탭, 아이콘+레이블, 활성=흰색·비활성=회색, 좌측에 백 버튼 |
| Inline cross-sell | 거래내역 row 사이에 끼워넣는 화살표+텍스트 prompt |

### 1.5 아이코노그래피

- 카테고리 아이콘은 **이모지처럼 보이는 컬러풀한 3D/스티커 풍** — 그러나 일관된 라이팅·각도. 이모지 시스템과 자체 아이콘 시스템의 경계가 흐릿하게 의도됨(친근감).
- 시스템 액션(검색/메뉴/체브론/하트)은 **모노톤 라인 아이콘**으로 통일 — 정보 vs 액션의 시각 분리.

## 2. 정책·의도·철학(추정)

### P1. "숫자가 주인공이다"
잔액·지수·가격을 화면 최대 글자로 박는다. 차트나 그래픽으로 미화하지 않고 **숫자 자체를 hero**로 둔다. 금융앱이지만 "데이터 콘솔" 같은 진지함을 유지.

### P2. "라이브 컨텍스트는 디폴트"
모든 섹션 타이틀 옆에 그 섹션과 관련된 **실시간 지수**가 인라인으로 붙는다. 증권=나스닥, 거래대금=VIX, 관심=코스피, 발견=코스닥, 피드=나스닥. 사용자는 어느 탭에 있든 "지금 시장이 어떤지" 알 수 있음. **탭과 지표의 매핑이 의도된 규칙**.

### P3. "개인화는 마케팅 카피가 아니라 골조"
"{이름}님"이 타이틀, 빈 상태, 섹션 헤더 어디든 들어간다. 빈 상태조차 이름 호명+CTA로 변환. 개인화를 **장식이 아닌 UI 컴포넌트의 슬롯**으로 취급.

### P4. "Dark-first, not dark-mode"
라이트의 다크 포팅이 아닌 다크에서 시작된 시각언어. 흰 글자에 의존하는 비중이 높고, 그라데이션·블러·일러스트가 다크 배경을 전제로 디자인됨. 라이트로 가면 오히려 어색해질 디자인 결정들이 많음.

### P5. "하나의 파랑으로 다 한다"
1차 CTA, 링크, 활성 탭, 랭킹 숫자, 하락 컬러를 모두 같은 브랜드 블루로 처리. **컬러를 의미별로 쪼개지 않고**, 컨텍스트(아이콘 모양·위치·문구)로 의미를 분리. 학습비용 ↓, 시스템 일관성 ↑, 다만 색맹/저시력 사용자에겐 까다로움.

### P6. "AI를 별도 채널로 분리"
보라 + ✨ + outlined 카드. **콘텐츠 vs AI 생성물의 시각적 출처 표시**. 신뢰 issue를 디자인으로 격리. AI가 만든 줄/말한 줄이 시각적으로 즉시 식별됨.

### P7. "광고는 숨기되 라벨링은 정직하게"
시각적 톤은 일반 카드와 동일하지만 "AD"/"서비스·광고"를 명확히 표기. 가독성을 깨지 않으면서 규제·신뢰를 충족하는 타협.

### P8. "이동을 줄이는 인라인 액션"
- 잔액 카드 안에 "이자 받기"가 박혀 있음.
- 거래내역 사이에 "안 쓰는 잔액 찾기" cross-sell.
- 차트 리스트 row 우측에 ❤️ 즉시 추가.

"다음 화면으로 가서 하는" 마찰을 최대한 깎아냄.

### P9. "빈 상태는 기능이다"
이모지 + 호명 + 1줄 본문 + 1개 CTA. 빈 상태가 **온보딩의 한 슬롯**으로 동작.

### P10. "묶기(grouping)는 시각 계층의 1차 무기"
서브헤딩 + 라운드 컨테이너 + 안쪽 row들. 설정/계좌상세에서 명확. 인지 청크 단위를 정렬.

### P11. "스크롤은 상단을 변환한다"
거래대금 화면에서 스크롤이 내려가니 상단 지수가 나스닥 → VIX로 바뀜. **타이틀 영역이 컨텍스트 변환 슬롯**으로 동작. 상단 영역을 정적 헤더가 아니라 **상태 디스플레이**로 본다.

### P12. "한국 금융 관행 존중"
- 빨강=상승, 파랑=하락 (서구 반대)
- `원` 후치, 천단위 콤마, 소수 없음
- "~님" 호칭

글로벌 시스템을 베껴 오지 않고 **로컬 컨벤션에 맞춰 토큰을 잡음**.

## 3. 재구성 시뮬레이션 — 추출한 규칙만으로 다시 만들 수 있나

각 화면을 **추출 규칙으로만** 풀어쓴 시뮬레이션 결과.

### 3-1. 증권 홈

```
[Header]
  좌: Title("토스증권") + 인라인 라이브 컨텍스트(나스닥 지수, +0.8%, 상승=red)
  우: AI ✨ · 검색 · 메뉴
[Promo Card · radius 20]
  중앙 일러스트(돈다발 3D)
  Title(2줄, Bold)
  Sub(회색)
  Primary CTA(블루 pill)
  우상단 X
[Secondary Card]
  좌: 텍스트 / 우: chevron — 채움 없는 surface 카드
[Section Header] "{이름}님을 위한 맞춤형 서비스" (P3)
[List rows × N]
  이모지풍 아이콘 / 라벨 / chevron
[Bottom Tab Bar] 증권/관심/발견/피드 + 좌측 백 버튼
```

미확정: 위쪽이 어떤 그라데이션을 쓰는지, 스크롤 시 변하는지는 정적 스샷으로 미확정.

### 3-2. 거래대금 차트

```
[Header (스크롤 변환됨)] VIX 17.26 -3.4% ← P11 적용
[AD Card] 라벨 "· AD" + 일반 카드 톤 ← P7
[Section] "실시간 거래대금 차트"
[Ranking rows × N]
  좌: 큰 블루 숫자(순위) ← P5
  중: 원형 로고 + 종목명 + 가격 + 델타(파랑/빨강) ← P12
  우: ❤️ (toggle) ← P8
[Link row] "다른 차트 보기" + chevron
[Pill toggle] "간편 홈 보기" — 모드 스위치
[Footer] 면책 문구(tertiary)
```

거의 100% 재구성 가능.

### 3-3. 관심(빈 상태)

```
[Header] Title("관심") + 코스피 지수 ← P2
[AI banner] ✨ "토스증권 AI 시그널" + 헤드라인 1줄 ← P6
[Inline tab chips] 최근 본 / 그룹추가
[Empty State]
  중앙 이모지(🔍)
  Title("{이름}님이 궁금했던 종목을 찾아보세요") ← P3, P9
  Sub("최근 본 종목이 없어요")
  Primary CTA("찾아보기")
```

완전 재구성. 빈 상태 규칙(P9)이 매우 강력하다는 검증.

### 3-4. 발견

```
[Header] Title("발견") + 코스닥
[Category chips × 4+ horizontal] 이모지 아이콘 카드
[Event widget] 좌: 텍스트 / 우: 코스피 — 인라인 라이브
[AI Card outlined] ✨ "AI 실시간 이슈" + 캐러셀 번호/제목 ← P6
[Section] "실시간 차트"
[Segmented tabs] 거래대금/거래량/급상승/급하락/인기 — 활성=밑줄
[Ranking rows × N] (3-2와 동일 컴포넌트)
```

컴포넌트가 3-2와 거의 재사용됨 → 시스템 일관성 검증 OK.

### 3-5. 피드

```
[Header] Title("피드") + 나스닥
[Segmented chip tabs] 전체(active chip) / 팔로잉 / 뉴스 / 콘텐츠 / 우측 프로필 아이콘
[Post]
  아바타 / 작성자명 / 뱃지(녹색 small pill, "5억대 자산가") / "> 토픽" / "팔로우"(블루 텍스트)
  메타("15시간 전(수정됨)")
  본문: 첫 줄 Bold ← 타이포 미세 규칙
  이미지(라운드 16) + 우하단 ⊕ floating
```

"뱃지+토픽+팔로우 한 줄"은 매우 정보밀도 높은 패턴. 추출 규칙으로 도달 가능.

### 3-6. 전체 메뉴

```
[Header] 이름 / 우측 상단 "고객센터 | 설정"
[Search bar] full-width, surface 카드형
[Section] "바로가기"
[Grid × 4] 큰 아이콘(folder형 포함) + 라벨
[Section] "최근 서비스" → 단일 아이콘
[Section] "금융 서비스"
[List rows] 큰 아이콘 + 라벨 + 우측 메타(점·구분) + chevron
[Bottom Tab Bar] 홈/혜택/쇼핑/증권/전체 ← 슈퍼앱 5탭, 현재 "전체" 활성
```

"금융서비스 row의 우측 메타"가 새로운 규칙(서브 카테고리를 점으로 나열). 정책 추가: **row 우측 메타는 점·구분으로 sub-options 노출**.

### 3-7. 설정

```
[Profile card] 아바타 + 이름 + sub("내 정보 · 주소 관리") + chevron
[Group Card · 일반]
  "언어 / 알림 / 화면 테마·진동 / 연락처 관리" — chevron rows
  "언어"만 우측에 현재값(한국어, 블루) ← P5 (블루=링크/현재 선택)
[Group Card · "인증 및 보안"]
[Group Card · "자산 및 증명서"]
```

**그룹 헤딩은 카드 바깥, 항목은 카드 안** 규칙이 뚜렷.

### 3-8. 통장 메인

```
[Top action] 좌 백 / 우 "카드, 관리"
[Hero block]
  계좌명+번호(작게, 회색 + underline 텍스트링크 느낌)
  잔액(초대형 Bold) ← P1
  하위 카드: "864원 이자 받기" Primary CTA inline ← P8
  최근 거래 3줄(날짜 약식 / 라벨 / 우측 금액 — 빨강 강조 없이 흰색 음수)
  하단 듀얼 CTA "채우기 / 보내기"
[Card] 자동이체 8개 chevron
[Card] "오늘 받을 수 있는 혜택 최소 877원" + "보기" ← P8
[Card] 용돈 0원 + 칩(+ 비상금 / + 여행비 / + 직접입력)
[Card] 미국 달러 1.00 USD + 환율 델타
```

**듀얼 CTA는 항상 페어**(채우기/보내기) 규칙 검증.

### 3-9. 거래내역

```
[Header] 백 / 카드 / 관리
[Hero] 계좌번호 underline / 잔액 중앙 정렬(초대형) / 풀-와이드 "+864원 이자 받기" pill(블루 outline 느낌)
[Filter row] "전체 ⌄" chip / 우측 검색 아이콘
[Group Header] "토스인컴 서비스·광고" ← P7
[Ad row] 큰 아이콘 + 2줄 텍스트 + chevron
[Date group] "5월 4일"
  Row: 아이콘 / 이름+시각 / 우측 금액+잔액
  Inline cross-sell card: ➡️ "투자 계좌 여러 개라면 안 쓰는 잔액 찾기" ← P8
[Sticky bottom CTA] 듀얼: 채우기(secondary) / 보내기(primary 블루)
```

**inline cross-sell은 거래 row 사이 슬롯에** 규칙 검증.

### 3-10. 계좌 상세

```
[Header] 백 / 우측 "내 계정"
[Title block] 큰 계좌별명 / sub "계좌번호·{번호}" / 우측 "복사" pill
[Info rows] 아이콘 + 라벨 + 우측 값 + chevron
[Section heading + group card] "내 계좌 관리" 외부 헤딩
[Section heading + group card] "이체 관리" — 자동이체 우측 "6건"이 블루(P5: 현재값/링크)
```

## 4. 정합성 검증 — 빈틈 / 가설로 남는 것

추출한 규칙으로 **10/10 화면 모두 골격 재구성 가능**. 그러나 정적 스샷이라 다음은 가설로 남음.

| 영역 | 미확인 | 검증 방법 |
|---|---|---|
| 모션 | 헤더 컨텍스트 전환(나스닥→VIX)이 페이드인지 슬라이드인지 | 동영상 또는 직접 사용 |
| 햅틱 | CTA·토글 햅틱 패턴 | 실기기 |
| 스크롤 시 hero shrink | 잔액 헤더가 sticky 축소되는지 | 실기기 |
| 라이트 모드 존재 여부 | 전부 다크 — 라이트 토큰 매핑이 있는지 | 설정 진입 |
| 접근성 | brand/blue 단일 사용의 색맹 보완(아이콘 형태?) | 코드 또는 a11y 감사 |
| 폰트 | Pretendard로 추정 — 실제 라이선스/대체 폰트 스택 | 실제 빌드 inspect |
| 그리드 | 컬럼 시스템(4/6/8 컬럼?)·기본 단위(4pt/8pt) | 정밀 픽셀 측정 |
| 일러스트 톤 | 3D 스티커풍의 라이팅/머티리얼 규칙 | 일러스트 가이드 |

**반례**: 거래내역 hero의 잔액이 *중앙 정렬*인데 통장 메인 hero는 *좌측 정렬*. → "**진입 깊이가 깊을수록(상세 페이지) 중앙 정렬로 hero를 더 격상시킨다**"는 추가 정책으로 해석 가능. 가설.

## 5. 종합 평가

**강점**

- 컴포넌트 재사용률이 매우 높음 (랭킹 row, 그룹 카드, 듀얼 CTA, 빈 상태 템플릿). 디자인 시스템이 잘 정착됨.
- "라이브 컨텍스트 + 개인화"를 화면 골조로 박은 게 결정적 차별점. 마케팅이 아닌 IA 차원의 결정.
- 단일 블루의 과부하를 **컨텍스트로 의미를 분리**해 우회 — 모험적이지만 일관성으로 정당화.

**약점/리스크**

- 단일 브랜드 컬러 의존 → 접근성 보조장치(아이콘/형태/굵기)에 추가 부하.
- "AD를 일반 카드 톤으로 통합" 정책은 규제 환경 변화에 취약.
- 정보 밀도가 높은 row(피드의 작성자+뱃지+토픽+팔로우)는 학습 곡선 존재.

**모방 시 주의**

- 단순히 색·폰트·라운드를 베끼면 **P1(숫자가 주인공) · P2(라이브 컨텍스트) · P3(개인화가 골조)** 이 빠져 "토스 같지 않은 토스 클론"이 됨. 진짜 차별점은 **데이터의 시각적 우선순위 결정 규칙**이지 시각 스타일이 아님.
- 토스를 베끼지 않고 "토스가 잘하는 것"만 가져가려면: **(1) hero 데이터 정의 → (2) 그 옆에 항상 붙는 라이브 컨텍스트 정의 → (3) 빈 상태를 컴포넌트화 → (4) 단일 액션 컬러로 신호 통일**.

## 6. 한 줄 결론

추출한 12개 정책(P1~P12)이 10개 화면 전체의 골조를 설명하고 재구성 가능함을 확인. 정적 스샷의 한계로 모션·햅틱·a11y 영역은 가설로 남김.

### 7.1 토스 공식: TDS(Toss Design System)

- **존재 이유**: TDS는 "토스 커뮤니티 전반에서 사용하는 공통의 디자인 언어이자 개발자의 도구"로 정의된다. 디자이너·개발자·기획자가 **같은 기준으로 협업**할 수 있도록 만든 인프라이며, 최소 품질을 보장하고 디자이너가 *"제품 자체에만 집중"* 하도록 돕는 것이 목적이다.
- **컬러 시스템(2024 전면 개편, 7년만)**: 단순 HEX가 아니라 **OKLCH/HSLuv 같은 인지적으로 균일한 색공간**을 기반으로 명도 시스템을 먼저 구축. 라이트모드와 다크모드의 명도 기준이 본질적으로 다르다는 점을 인정하고, **다크모드에 더 강한 명도 대비**를 부여. 라이트/다크는 1:1 자동 매핑.
- **시맨틱 토큰**: 색상의 값(Primitive)이 아니라 **사용 의도(Semantic)** 에 집중한 토큰 체계(`fill`/`text`/`border` × `brand`/`neutral`/`primary`/`secondary`). 동일한 의도면 같은 색이 나오게 강제하여 일관성 보장.
- **자동화**: Figma의 Token Studio → GitHub → 모든 플랫폼 코드 자동 생성. 디자인-구현 사이의 동기화 비용 제로화.
- **컴포넌트 결정 원칙**: 컴포넌트는 *"왜 존재하는가"* 가 합의되어야 추가된다. 즉 TDS는 **실험과 논증을 통과한 컴포넌트만 살아남는다**.

### 7.2 토스 공식: UX Writing 코어밸류 + 라이팅 원칙

토스는 **카피도 디자인 시스템의 일부**로 본다. 다만 공개된 자료를 보면 **보이스톤을 정의하는 코어밸류**와 **실무 라이팅 원칙(8가지 라이팅 원칙)**은 서로 다른 층위이므로 분리해서 정리한다.

**(a) 보이스톤 코어밸류** — 토스 카피의 "성격"을 규정:

| 코어밸류 | 한 줄 |
|---|---|
| **Clear** | 사용자가 한 번에 이해 가능한 단어/문장만 사용. 메이커 관점 금지. |
| **Concise** | 사용자는 안 읽는다는 전제. 필요한 만큼만, 스캔 가능하게. |
| **Casual** | 친근한 보이스톤을 **토스 정체성의 핵심 차별점**으로 본다. |

(공개 자료상 코어밸류는 다섯 개로 언급되나 외부 공개된 항목은 위 세 개이며, 나머지 두 개는 본 디스커버리 시점에서 미확인.)

**(b) 8가지 라이팅 원칙** — 코어밸류를 실무 카피로 옮기는 가이드라인. 공개 자료에서 확인된 항목 일부:

| 원칙 | 한 줄 |
|---|---|
| **User-side Info** | 정보를 유저 입장에서 재구성해서 제공. |
| **Weed Cutting** | 의미 없는 문장은 잡초처럼 제거. |

(나머지 6개 원칙은 본 디스커버리 시점의 공개 자료에서 전부 확인되지 않음. 원문 확인 시 보강 필요.)

### 7.4 외부 분석: "한 화면 한 기능" 원칙

여러 외부 분석가가 토스 UX를 한 마디로 요약할 때 가장 자주 인용하는 표현. 한 화면에는 사용자의 **단일 결정**만 요구하고, 보조 정보·교차판매·옵션은 다음 화면이나 인라인 슬롯으로 미룬다.

| P2 라이브 컨텍스트 디폴트 | △ | 공식 디자인 토큰 차원 언급 미발견. 본 분석의 **추측이 외부에서 검증되지 않은 가설**. |
| P3 개인화는 골조 | ◯ | UX Writing의 *User-side Info* **라이팅 원칙**(7.2.b)과 정합. 단 "구조화된 슬롯"으로 명시한 자료는 못 찾음. |
| P4 Dark-first | ◎ | TDS 컬러 업데이트 글이 라이트/다크 명도 기준의 본질적 차이를 명시. 강하게 일치. |
| P5 하나의 파랑 | ◎ | 시맨틱 토큰 = "동일 의도면 같은 색"과 정합. |
| P6 AI 별도 채널 | △ | 공식 명문화는 없으나 시맨틱 토큰 체계상 가능한 결정. |
| P7 광고 라벨링 정직 | ─ | 디자인 자료에서 직접 언급 못 찾음. 규제 컨벤션일 가능성. |
| P8 인라인 액션 | ◯ | "한 화면 한 기능"과 동일한 방향(이동 줄이기). |
| P9 빈 상태는 기능 | ◯ | *Casual* + *Clear* **코어밸류**(7.2.a)의 자연스러운 귀결. |
| P10 묶기(grouping) | ◎ | TDS 컴포넌트 가이드 작성 원칙: "구조 → 요소" 순서. 직접 일치. |
| P11 스크롤이 상단을 변환 | ─ | 외부 자료에서 매핑 못 찾음. **본 분석 고유 관찰**. |
| P12 한국 금융 관행 존중 | ◯ | 산업 표준이지 토스 고유는 아님. |

◎ 강한 일치 / ◯ 일치 / △ 부분 일치·추정 / ─ 미확인

### 7.8 외부 자료가 알려준 **새 정책**(분석에서 빠뜨린 것)

- **P0 "한 화면 한 기능"**: P1~P12를 관통하는 *상위 원칙*. 본 분석은 이 원칙의 *결과*만 봤지 원칙 자체를 식별하지 못함. 다음 분석부터는 이걸 0번으로 두고 P1~P12를 *전술*로 본다.
- **P13 "라이팅은 디자인 시스템의 일부"**: 카피·보이스톤도 토큰처럼 관리됨. UI 톤만 보고 라이팅을 분리하던 본 분석의 가정은 부정확.
- **P14 "디자인 토큰의 자동화 파이프라인"**: Figma → GitHub → 멀티플랫폼 자동 생성. 토큰을 사람이 옮기는 비용을 제로화. 본 분석은 토큰을 추출했지만 **운영 메커니즘**까지는 못 봤음.
- **P15 "컴포넌트는 논증을 통과해야 산다"**: 즉 디자인 시스템 자체가 **자생적 가지치기 시스템**. 컴포넌트 다양성보다 정당성을 우선.

### 7.9 정합성 재검증 결론

- P4, P5, P10은 **외부 공식 자료로 강하게 검증됨** → 확신도 상향.
- P2, P11은 **외부 자료에서 매핑 못 찾음** → 본 분석의 가설로 명시 유지.
- **P0(한 화면 한 기능), P13~P15** 4개 정책을 새로 추가하면, 추출 모델이 12개 → 16개로 확장되며 토스 디자인의 *방법론적 구조*(원칙 → 시스템 → 자동화 → 거버넌스)까지 포괄.

## 8. 병행 분석 — GPT-5 (스크린샷 단독)

> 본 섹션은 동일 스크린샷을 입력으로 GPT-5가 별도로 수행한 분석을 원문에 가깝게 보존한 것이다. 1~7장(Claude 분석)과 결론이 일부 겹치고 일부 다르므로 **교차 검증용 별도 기록**으로 둔다. 헤더 레벨만 문서 구조에 맞춰 조정했고, 본문은 가공하지 않았다. 각주 ID는 본 장 한정 `[g*]`로 부여한다.

### 8.0 도입

아래 분석은 **"토스 UI를 베끼기 위한 명세"가 아니라, 토스가 해결한 UX 문제와 설계 원리를 추출한 것**으로 보는 게 안전하다. UI/UX는 아이디어·패턴·기능적 구조와 구체적 시각 표현이 섞여 있어서, 원리는 참고하되 **색, 곡률, 아이콘, 문구, 배치 순서, 컴포넌트 조합, 애니메이션, 브랜드 톤**은 독자적으로 재설계해야 한다. 저작권 쪽에서도 일반적으로 아이디어 자체가 아니라 창작적 "표현"이 보호 대상이고, UI 디자인은 저작권·디자인권·상표/트레이드드레스 등 여러 보호 영역이 겹칠 수 있는 복합 대상이다. ([Copyright Commission of Korea][g1])

또한 스크린샷에 개인 이름, 계좌, 잔액 등이 보이므로 외부 공유용 문서나 프롬프트에는 반드시 마스킹하는 게 좋다.

### 8.1 한 줄 결론

이 UI의 핵심은 "예쁜 다크모드 카드 UI"가 아니다.

핵심은 다음 5가지다.

1. **복잡한 금융 기능을 '작은 카드/행/칩' 단위로 분해한다.**
2. **모든 화면을 '지금 사용자가 할 법한 다음 행동' 중심으로 재배열한다.**
3. **정보보다 행동을 앞세우되, 금융 신뢰를 해치지 않도록 숫자·고지·상태를 숨기지 않는다.**
4. **홈, 증권, 피드, 전체 메뉴, 계좌 상세가 서로 다른 제품처럼 보이지 않게 동일한 문법으로 묶는다.**
5. **유틸리티 앱과 콘텐츠/커뮤니티 앱 사이를 오가며 사용 빈도를 높인다.**

즉, 토스의 철학은 대략 이렇다.

> "금융은 어렵고 드문 일이 아니라, 매일 확인하고 가볍게 조작하는 개인 생활 OS다."

### 8.2 화면별 구조 해체

#### 8.2.A 토스증권 홈

첫 화면은 전형적인 금융 대시보드가 아니다. 보통 증권 앱이라면 보유 종목, 평가손익, 주문, 관심종목, 차트가 먼저 나와야 한다. 그런데 여기서는 다음 순서다.

1. 상단: 브랜드/현재 시장 지표
2. 중앙: 계좌개설 이벤트 카드
3. 그 아래: 계좌 만들기 CTA
4. 개인화 서비스 리스트
5. 하단: 증권/관심/발견/피드 탭

즉 이 화면은 "이미 증권을 쓰는 사람"보다 **증권 계좌를 아직 만들지 않은 사용자**를 대상으로 한다. 화면의 실제 목적은 정보 제공이 아니라 **증권 계좌 개설 전환**이다.

숨은 정책은 이렇다.

| 관찰 | 숨은 의도 |
|---|---|
| 계좌개설 이벤트가 최상단 대형 카드 | 핵심 비즈니스 전환을 첫 화면의 주인공으로 둠 |
| "문제 보기", "계좌 만들기"처럼 쉬운 언어 사용 | 금융 진입장벽을 퀴즈/이벤트로 낮춤 |
| "맞춤형 서비스" 섹션 | 사용자가 비어 있는 상태에서도 할 일을 제공 |
| 나스닥 지표를 제목 옆에 배치 | 사용자가 증권 화면에 들어왔다는 시장 맥락 제공 |
| 하단 탭이 둥근 플로팅 바 | 앱 내부의 독립 제품처럼 느끼게 함 |

토스증권 홈은 "증권 기능 목록"이 아니라 **온보딩 + 리텐션 + 크로스셀 화면**이다.

#### 8.2.B 실시간 거래대금 차트 화면

두 번째 증권 화면은 스크롤 후 영역으로 보인다. 여기서는 광고, 실시간 거래대금 차트, 다른 차트 보기, 간편 홈 보기, 투자 유의문구가 배치되어 있다.

핵심은 "랭킹"이다. 사용자는 복잡한 차트보다 "지금 뭐가 움직이나?"를 먼저 알고 싶어 한다. 그래서 개별 종목 분석보다 **실시간 랭킹 콘텐츠**를 앞세운다.

숨은 정책:

| 관찰 | 해석 |
|---|---|
| 거래대금 차트가 1~3위만 노출 | 전체 목록보다 즉시 이해 가능한 스냅샷 제공 |
| 종목 로고를 원형 아이콘으로 제공 | 금융 정보를 소비자 앱의 리스트처럼 부드럽게 만듦 |
| 하락률은 파란색, 상승률은 빨간색 | 한국 주식 관습을 반영 |
| 하트 아이콘 | "관심종목 추가"를 별도 기능이 아니라 리스트 행의 기본 액션으로 만듦 |
| 하단 고지문 | 투자 정보 제공과 투자 권유 사이의 책임 경계 설정 |

좋은 점은 랭킹, 로고, 가격, 등락률, 관심 추가가 한 줄 안에서 깔끔하게 정리된다는 것이다. 아쉬운 점은 "거래대금"이라는 기준이 초보자에게는 위험할 수 있다는 점이다. 거래대금 상위 종목은 보통 과열, 급락, 테마, 변동성과 연결되기 쉽다. 따라서 토스는 접근성을 높였지만, 동시에 **투자 행동을 자극하는 UX**도 강하다.

#### 8.2.C 관심 탭의 빈 상태

관심 화면은 최근 본 종목이 없는 상태다. 중앙에 돋보기 일러스트, 짧은 문구, CTA가 있다.

이 화면의 핵심은 "빈 화면을 실패처럼 보이지 않게 만드는 것"이다.

| 요소 | 의미 |
|---|---|
| 큰 빈 공간 | 사용자를 압박하지 않음 |
| "궁금했던 종목을 찾아보세요" | 사용자의 과거 행동 없음 = 문제로 표현하지 않음 |
| "최근 본 종목이 없어요" | 상태를 정확히 설명 |
| "찾아보기" CTA | 다음 행동을 하나만 제안 |
| 상단 AI 시그널 | 빈 상태에서도 시장 맥락은 유지 |

이 빈 상태는 잘 설계되어 있다. 다만 앱이 너무 콘텐츠 지향적으로 흐르면 관심 목록이 비어 있을 때도 "AI 시그널", "뉴스", "이슈"가 사용자를 매수/매도 쪽으로 유도할 수 있다. 금융 앱에서는 빈 상태조차 중립성이 중요하다.

#### 8.2.D 발견 탭

발견 탭은 사실상 증권 앱의 "탐색/콘텐츠 허브"다.

구조는 이렇다.

1. 제목 + 시장 지표
2. 자산군 카테고리 칩: 국내주식, 해외주식, 옵션, 채권 등
3. 예정 이벤트
4. 코스피 요약
5. AI 실시간 이슈
6. 실시간 차트 탭
7. 종목 랭킹 리스트

이 화면은 명확히 **"투자자가 뭘 봐야 할지 모르는 상태"**를 해결한다.

숨은 정책:

| 관찰 | 해석 |
|---|---|
| 자산군을 카드형 칩으로 배치 | 메뉴가 아니라 탐색 콘텐츠처럼 보이게 함 |
| "6일 뒤 이벤트" | 시간축을 넣어 시장을 캘린더화 |
| "AI 실시간 이슈" | 복잡한 뉴스를 하나의 요약 진입점으로 압축 |
| 차트 탭: 거래대금/거래량/급상승/급하락/인기 | 사용자의 투자 관심사를 랭킹 축으로 번역 |
| 코스피 수치가 반복 노출 | 화면 전역에 시장 분위기 유지 |

이 화면의 강점은 "금융 초보자도 어디부터 볼지 알게 해준다"는 점이다. 약점은 발견/랭킹/AI 이슈가 결합되면 **FOMO 유발 UI**가 되기 쉽다는 점이다. 실제 서비스 설계에서는 "왜 이 종목이 노출되었는지", "위험도", "변동성 경고" 같은 보조 장치가 중요하다.

#### 8.2.E 피드 탭

피드는 금융 앱이라기보다 소셜/커뮤니티 피드에 가깝다. 사용자 닉네임, 태그, 종목명, 팔로우, 본문, 이미지, 플로팅 액션이 있다.

이 화면의 숨은 의도는 명확하다.

> 증권 앱을 "주문할 때만 여는 앱"에서 "매일 읽는 앱"으로 바꾸는 것.

좋은 점:

- 체류시간 증가
- 사용자 생성 콘텐츠 기반 리텐션
- 관심 종목과 커뮤니티를 연결
- 금융 정보를 감정적으로 소비하게 만듦

위험한 점:

- 과장, 선동, 확증편향, FOMO가 강해질 수 있음
- 투자 앱의 신뢰성과 커뮤니티의 자극성이 충돌할 수 있음
- "5억대 자산가" 같은 배지는 권위 효과를 만들 수 있음
- 초보자는 콘텐츠와 투자 조언을 구분하기 어려움

피드는 UX적으로 강력하지만, 금융 서비스 관점에서는 정책/모더레이션/리스크 고지가 매우 중요하다.

#### 8.2.F 전체 메뉴

전체 메뉴는 토스 본체의 서비스 포털이다. 증권 앱과 다르게 계좌, 대출, 요금제, 신분증, 토스뱅크, 게임, 미니앱 등이 같이 있다.

핵심은 **슈퍼앱 구조**다.

| 관찰 | 해석 |
|---|---|
| 상단 검색 | 서비스 수가 많기 때문에 탐색보다 검색을 우선 |
| 바로가기 아이콘 4개 | 사용 빈도 높은 기능을 단축 |
| 최근 서비스 | 사용자의 실제 행동 기반으로 개인화 |
| 금융 서비스 리스트 | 전체 기능을 카테고리화 |
| 하단 탭: 홈/혜택/쇼핑/증권/전체 | Toss 전체 앱의 최상위 네비게이션 |

여기서 중요한 점은 "전체 메뉴"가 서비스 목록임에도 지나치게 관리형/admin스럽지 않다는 것이다. 각 항목은 텍스트와 아이콘, 부가 설명만으로 구성되어 있고, 기능은 많지만 무겁게 보이지 않는다.

#### 8.2.G 설정 화면

설정 화면은 카드형 그룹으로 나뉜다.

1. 프로필
2. 일반 설정: 언어, 알림, 화면 테마, 연락처
3. 인증/보안
4. 자산 및 증명서
5. 송금 및 결제

이 화면은 토스 UI의 매우 중요한 특성을 보여준다.

> 토스는 "복잡한 기능"을 화면마다 숨기는 것이 아니라, **그룹 단위로 부드럽게 감싼다.**

설정 화면은 금융 앱 치고 항목이 많지만, 카드 그룹 사이 여백이 크고, 각 행의 높이가 넉넉하다. 그래서 "설정 지옥"처럼 보이지 않는다.

#### 8.2.H 토스뱅크 계좌 홈

계좌 화면은 증권보다 훨씬 더 "금융 본질"에 집중한다.

구조:

1. 계좌명/계좌번호/잔액
2. 이자 받기
3. 최근 거래 요약
4. 채우기/보내기
5. 자동이체
6. 받을 수 있는 혜택
7. 용돈/목표성 지출
8. 외화

여기서는 카드 UI가 단순한 장식이 아니다. 계좌라는 복잡한 객체를 다음과 같은 하위 객체로 분리한다.

- 잔액
- 이자
- 거래내역
- 송금/입금
- 자동이체
- 혜택
- 예산/용돈
- 외화

좋은 점은 사용자가 계좌를 볼 때 가장 필요한 행동인 **보내기/채우기**가 매우 명확하다는 것이다. 금융 앱에서 중요한 "주요 행동 2개"만 크게 남긴다.

#### 8.2.I 계좌 상세 거래내역

거래내역 화면은 상단에 잔액을 크게 고정하고, 이자 CTA를 큰 파란 영역으로 보여준다. 그 아래 필터, 검색, 광고/안내, 날짜별 거래내역, 하단 고정 CTA가 있다.

이 화면의 핵심은 **스크롤 가능한 데이터 화면 + 고정 액션**이다.

| 요소 | 의미 |
|---|---|
| 잔액 대형 표시 | 사용자가 이 화면에 온 목적을 즉시 충족 |
| 이자 CTA | 계좌의 핵심 혜택을 반복 노출 |
| 필터/검색 | 거래내역 탐색 지원 |
| 날짜 그룹 | 회계/가계부식 이해를 돕는 구조 |
| 하단 채우기/보내기 고정 | 데이터를 보다가 즉시 행동 가능 |

이 패턴은 가계부 앱에도 매우 유용하다. 월별 지출, 카테고리, 거래내역을 보여주되 하단에 "기록하기/송금하기/정산하기" 같은 행동을 고정할 수 있다.

#### 8.2.J 계좌 관리

계좌 관리 화면은 가장 정보 밀도가 높지만 여전히 단순하다.

구조:

1. 계좌명, 계좌번호, 복사
2. 금리, 체크카드
3. 내 계좌 관리
4. 이체 관리

여기서 눈에 띄는 철학은 **상세 설정은 평면 리스트로, 하지만 그룹 단위로 분리**한다는 것이다. 버튼을 남발하지 않고, 오른쪽 값과 chevron으로 상태와 이동을 동시에 표현한다.

### 8.3 디자인 시스템 추출

#### 8.3.1 색상 정책

정확한 색상값을 복제하면 위험하므로, 원리는 이렇게 추출하는 게 맞다.

| 역할 | 토스 화면의 기능 | 독자 설계 시 권장 |
|---|---|---|
| Base background | 거의 검정에 가까운 남색/회색 | 순흑색보다 약간 밝은 다크 네이비/차콜 |
| Surface card | 배경보다 한 단계 밝은 카드 | 1~2단계 elevation만 사용 |
| Primary action | 선명한 파란색 | 브랜드 고유의 액션 컬러 사용 |
| Positive market | 한국식 빨간색 | 서비스 국가/자산군 관습에 맞춤 |
| Negative market | 한국식 파란색 | 액션 블루와 혼동되지 않게 구분 필요 |
| Primary text | 거의 흰색 | 충분한 대비 |
| Secondary text | 회색 | 설명/보조정보용 |
| Divider | 매우 약한 회색 | 경계는 보이되 튀지 않게 |

핵심은 "색을 적게 쓰는 것"이다. 화면 대부분은 무채색이고, 파란색은 액션/링크/시장 하락/AI 시그널 등 일부 의미에만 쓰인다. 이 때문에 화면이 복잡해도 시각적으로는 조용하다.

다만 액션 파랑과 하락 파랑이 같은 계열이라, 독자 앱에서는 이 둘을 더 명확히 분리하는 편이 좋다.

#### 8.3.2 타이포그래피 정책

토스 UI는 글자가 크다. 특히 모바일 금융 앱에서 작은 글씨를 많이 쓰지 않는다.

추출 규칙:

| 텍스트 종류 | 역할 |
|---|---|
| 대형 타이틀 | 현재 화면의 정체성 |
| 시장 지표/잔액 | 화면의 핵심 상태 |
| 섹션 제목 | 정보 블록의 시작점 |
| 리스트 주 텍스트 | 기능/종목/거래명 |
| 보조 텍스트 | 가격, 시간, 설명, 태그 |
| 법적 고지 | 하단 저강도 텍스트 |

철학은 이렇다.

> "사용자가 읽어야 할 것은 크게, 읽어도 되고 안 읽어도 되는 것은 작고 흐리게."

이 원칙 덕분에 화면 내 정보량이 많아도 우선순위가 분명하다.

#### 8.3.3 간격과 레이아웃 정책

대체로 다음과 같은 리듬이 보인다.

| 항목 | 추정 원리 |
|---|---|
| 좌우 마진 | 넉넉한 20~24pt 계열 |
| 카드 내부 패딩 | 20~24pt 계열 |
| 카드 간격 | 12~20pt 계열 |
| 섹션 간격 | 32~48pt 계열 |
| 리스트 행 높이 | 터치 가능한 56~72pt 계열 |
| 버튼 높이 | 48~56pt 계열 |
| 하단 내비게이션 | 플로팅 pill 형태, 큰 터치 영역 |

이 UI는 촘촘하게 정보를 밀어 넣지 않는다. 특히 금융 앱인데도 "공백"을 많이 쓴다. 그 공백이 프리미엄함과 신뢰감을 만든다.

#### 8.3.4 카드 정책

카드는 토스 UI의 기본 문법이다. 하지만 모든 카드가 같은 역할은 아니다.

| 카드 유형 | 예시 | 역할 |
|---|---|---|
| Hero card | 이벤트, 계좌 요약 | 화면의 주인공 |
| Action card | 계좌 만들기, 이자 받기 | 행동 유도 |
| List card | 맞춤 서비스, 설정 그룹 | 기능 목록 |
| Data card | 거래대금 차트, 계좌 요약 | 숫자/상태 요약 |
| Empty card/state | 관심 없음 | 빈 상태 해결 |
| Feed card | 커뮤니티 게시물 | 콘텐츠 소비 |

중요한 것은 카드가 "예쁘게 둘러싸는 장식"이 아니라, **사용자의 다음 행동 단위**라는 점이다.

#### 8.3.5 아이콘 정책

토스 UI의 아이콘은 설명을 대체하기보다, 리스트 스캔 속도를 높인다.

- 종목 로고: 식별성
- 금융 서비스 아이콘: 기능 범주
- 하트: 관심 추가
- chevron: 이동 가능성
- 돋보기: 탐색
- 스파클: AI/추천/시그널
- 캘린더, 스쿠터, 불꽃 등: 서비스/이벤트 성격

아이콘의 역할은 "장식"이 아니라 **행의 의미를 0.2초 안에 파악하게 하는 것**이다.

### 8.4 UX 철학 추출

#### 8.4.1 "금융을 기능이 아니라 상황으로 보여준다"

토스는 "계좌", "증권", "대출", "송금" 같은 기능명을 나열하기도 하지만, 핵심 화면에서는 사용자의 상황을 먼저 보여준다.

예:

- 받을 수 있는 혜택
- 궁금했던 종목
- 최근 본 종목
- 오늘 받을 수 있는 혜택
- 6일 뒤 이벤트
- 실시간 이슈
- 안 쓰는 잔액 찾기

이런 문구는 기능 중심이 아니라 상황 중심이다.

독자 앱에 적용하면:

- "거래내역"보다 "이번 달 돈이 어디로 갔는지 보기"
- "예산 설정"보다 "이번 달 남은 생활비 확인"
- "카테고리 관리"보다 "자주 쓰는 지출 묶기"
- "통계"보다 "지난달보다 많이 쓴 항목"

#### 8.4.2 "한 화면에 하나의 대표 행동"

각 화면마다 대표 행동이 있다.

| 화면 | 대표 행동 |
|---|---|
| 증권 홈 | 계좌 만들기 / 이벤트 참여 |
| 관심 | 종목 찾아보기 |
| 발견 | 시장 이슈/랭킹 탐색 |
| 피드 | 콘텐츠 읽기/팔로우 |
| 전체 | 서비스 검색/실행 |
| 계좌 홈 | 보내기/채우기 |
| 계좌 상세 | 거래 확인 + 보내기/채우기 |
| 계좌 관리 | 설정 변경 |

토스의 강점은 이 대표 행동이 흔들리지 않는다는 점이다. 화면에 여러 기능이 있어도 CTA가 하나 또는 둘로 제한된다.

#### 8.4.3 "복잡함은 숨기지 않고, 늦게 보여준다"

토스는 기능을 삭제하지 않는다. 대신 첫 화면에서 보이지 않게 한다.

예:

- 계좌번호, 금리, 체크카드, 자동이체, 송금한도, ATM 한도는 모두 존재한다.
- 하지만 계좌 홈에서는 잔액/이자/보내기/채우기가 먼저다.
- 세부 설정은 계좌 관리로 이동해야 보인다.

이것은 단순화가 아니라 **progressive disclosure**다.

#### 8.4.4 "초보자에게 친절하지만, 완전히 유아화하지 않는다"

문구는 쉽다. 하지만 데이터는 실제다.

- 시장 지수
- 가격
- 등락률
- 거래대금
- 거래내역
- 잔액
- 계좌번호
- 금리

즉, "쉬운 말"과 "실제 금융 데이터"를 동시에 둔다. 이 균형이 중요하다. 너무 쉬운 말만 쓰면 신뢰가 떨어지고, 숫자만 쓰면 진입장벽이 높아진다.

#### 8.4.5 "개인화는 사용자를 붙잡는 기본 장치"

이름, 최근 서비스, 최근 본 종목, 맞춤형 서비스, 받을 수 있는 혜택이 반복된다.

개인화의 목적은 두 가지다.

1. 내가 이 앱 안에 이미 존재한다는 감각
2. 지금 나에게 할 일이 있다는 감각

이는 리텐션에 매우 강하다.

#### 8.4.6 "광고/마케팅도 제품 흐름 안에 넣는다"

화면에는 이벤트, 광고, 혜택, 계좌 개설, 서비스 추천이 많다. 하지만 대부분 카드/행 문법 안에 들어가 있어 노골적인 배너처럼 보이지 않는다.

이 방식의 장점:

- 광고 피로도가 낮다.
- 서비스 흐름을 끊지 않는다.
- 전환율이 높을 가능성이 크다.

위험:

- 금융 앱에서 광고와 정보의 경계가 흐려질 수 있다.
- 사용자가 추천/광고/콘텐츠/고지를 구분하기 어려울 수 있다.
- 신뢰를 잃지 않으려면 AD 표기, 출처, 이유, 위험 고지가 명확해야 한다.

### 8.5 숨은 제품 전략

#### 8.5.1 토스증권은 "브로커리지"가 아니라 "금융 미디어 + 거래 플랫폼"이다

증권 화면은 주문 중심이 아니다. 관심, 발견, 피드, AI 시그널, 실시간 이슈, 랭킹이 매우 강하다.

즉 토스증권은 사용자를 이렇게 이동시키려 한다.

> 시장 보기 → 이슈 보기 → 종목 발견 → 관심 추가 → 커뮤니티 소비 → 계좌 개설/거래

이것은 전통 증권사 앱보다 훨씬 콘텐츠 플랫폼에 가깝다.

#### 8.5.2 토스뱅크는 "계좌"를 "행동 가능한 카드"로 바꾼다

계좌 화면에서 사용자는 잔액만 보는 게 아니라 바로 행동한다.

- 이자 받기
- 채우기
- 보내기
- 자동이체 보기
- 혜택 받기
- 용돈 설정
- 외화 보기

계좌는 정적인 자산 객체가 아니라 **매일 조작하는 생활 객체**가 된다.

#### 8.5.3 전체 메뉴는 "금융 서비스 백화점"이 아니라 "개인 OS 런처"다

전체 메뉴는 토스의 슈퍼앱 전략을 보여준다. 홈, 혜택, 쇼핑, 증권, 전체가 하나의 앱 안에 묶여 있고, 검색과 최근 서비스가 이 복잡성을 완충한다.

서비스가 많아질수록 카테고리 메뉴보다 중요한 것은:

- 최근 사용
- 검색
- 개인화 추천
- 명확한 아이콘
- 쉬운 문구

이다.

### 8.6 좋게 평가할 부분

- **화면 목적이 분명하다**: 각 화면마다 사용자가 해야 할 일이 명확하다. "모든 기능을 다 보여주는 금융 앱"이 아니라, "지금 이 화면에서 필요한 행동"을 중심으로 설계되어 있다.
- **정보 밀도 대비 피로도가 낮다**: 실제 정보량은 많다. 지수, 종목, 계좌, 거래내역, 설정, 광고, 피드가 모두 있다. 그런데 다크 배경, 큰 여백, 카드 그룹, 약한 구분선 덕분에 피로도가 낮다.
- **빈 상태 설계가 좋다**: 관심 탭의 empty state는 사용자가 아무것도 하지 않은 상태를 자연스럽게 처리한다. 빈 화면을 방치하지 않고, 다음 행동을 하나만 제안한다.
- **기능과 콘텐츠의 연결이 강하다**: 종목 랭킹, AI 이슈, 피드, 관심 추가가 서로 연결되어 있다.
- **금융 앱인데 소비자 앱처럼 느껴진다**: 전통 금융 앱의 딱딱함을 줄이고, 일상 앱처럼 접근하게 만든다.

### 8.7 비판적으로 봐야 할 부분

- **투자 앱에서 FOMO를 만들 수 있다**: 실시간 차트, 급상승/급하락, 인기, 피드, AI 이슈는 모두 행동을 자극한다.
- **AI 시그널의 신뢰 경계가 불분명할 수 있다**: 출처, 신뢰도, 지연 시간, 요약 근거가 중요하다.
- **광고와 유틸리티의 경계가 흐려질 수 있다**: AD 표기가 작거나 흐리면 신뢰 리스크가 생긴다.
- **하단 네비게이션 계층이 많다**: 토스 전체 하단 탭과 증권 내부 하단 탭이 서로 다르다.
- **표면 디자인만 따라 하면 실패한다**: 진짜 핵심은 정보 구조, 행동 우선순위, 상태 설계, 개인화, 마이크로카피다.

### 8.8 추출한 설계 원칙

토스 UI에서 추출할 수 있는 재사용 가능한 원칙은 다음이다.

1. **화면은 데이터 단위가 아니라 사용자 과업 단위로 나눈다**
2. **첫 화면에는 "전체"가 아니라 "대표 행동"만 둔다**
3. **리스트 행은 정보와 행동을 동시에 담는다** (아이콘 / 주 텍스트 / 보조 텍스트 / 현재 상태값 / 이동 가능성 / 가끔 즉시 액션)
4. **빈 상태는 CTA가 있는 상태다**
5. **카드 그룹은 사용자의 심리적 부담을 줄인다**
6. **숫자는 숨기지 말고, 해석을 붙인다**
7. **개인화 문구는 기능보다 강하다** (단, 과도하게 사용하면 조작적 느낌)
8. **금융 앱의 커뮤니티/피드는 별도 안전장치가 필요하다** (신고, 출처, 투자권유 금지, 위험고지, 과장 표현 감지, 랭킹 노출 제한)

### 8.10 재구성 시뮬레이션

각 화면을 추출한 컴포넌트 + 원칙으로 다시 만들 수 있는지 시뮬레이션. 본 분석의 1~7장에서 이미 같은 작업을 했으므로 GPT 결과만 요약:

- **증권 홈** → `ContextHeader + HeroCard + ActionRow + GroupedListCard + FloatingBottomNav`로 재구성 가능. 핵심은 **전환 중심 hero + 개인화 리스트 + 하단 nav**.
- **거래대금 차트** → `AdRow + SectionHeader + RankingList + SecondaryCTA + DisclaimerBlock`. `RankingList`와 `DisclaimerBlock`만 있으면 구조 재현.
- **관심 빈 상태** → `ContextHeader + AISignalBanner + SegmentedTabs + EmptyState`. 핵심 원칙은 **빈 상태에도 맥락과 다음 행동을 제공한다**.
- **발견 탭** → `ChipScroller + MarketIssue + AISignalBanner + SegmentedRankingTabs + RankingList`.
- **피드** → `FeedFilterTabs + FeedPost + FloatingActionButton`. 다만 토스 고유의 콘텐츠 톤이 강해 그대로 따라 하면 모방 느낌이 가장 많이 남.
- **전체 메뉴** → `ProfileHeader + SearchBar + ShortcutGrid + RecentServices + ServiceList + GlobalBottomNav`.
- **설정** → `BackHeader + ProfileRow + SettingsGroupCard`. `GroupedListCard` 원칙의 순수한 사례.
- **계좌 홈** → `AccountSummaryCard + InterestCTA + RecentTransactionPreview + DualActionButtons + LinkedFeatureRows + MiniAssetCards`. 계좌를 "행동 가능한 객체"로 만드는 게 핵심.
- **거래내역 상세** → `StickyAccountHeader + LargeCTA + TransactionFilter + TransactionList + StickyBottomActions`. `데이터 리스트 + 고정 액션` 패턴.
- **계좌 관리** → `AccountInfoHeader + AttributeRows + ManagementGroupCard`. `상태값 있는 설정 리스트` 패턴.

검증 결과: 위 컴포넌트 + 원칙 세트만으로 스크린샷 대부분을 구조적으로 재구성 가능.

### 8.11 정합성 검증 결과

추출한 원칙으로 대부분의 화면을 설명할 수 있다. 따라서 분석의 정합성은 높다.

**설명 가능한 공통 패턴**

| 패턴 | 등장 화면 |
|---|---|
| 큰 제목 + 현재 상태 | 증권, 관심, 발견, 피드, 계좌 |
| 카드형 surface | 거의 전 화면 |
| 하단 플로팅 nav | 증권 내부 화면 |
| 그룹 리스트 | 전체 메뉴, 설정, 계좌 관리 |
| 대표 CTA | 계좌개설, 찾아보기, 이자 받기, 보내기 |
| 개인화 문구 | 홈, 전체, 계좌 |
| 랭킹/피드형 콘텐츠 | 증권 발견, 피드 |
| 빈 상태 + CTA | 관심 |
| 데이터 + 고정 액션 | 거래내역 상세 |

**추가로 필요했던 예외 규칙**

처음 추출한 원칙만으로는 부족하고, 다음 규칙을 추가해야 전체 재구성이 가능하다.

1. 금융 앱이지만 콘텐츠 피드 문법을 허용한다.
2. 광고/추천/혜택은 제품 흐름 안에 통합한다.
3. 슈퍼앱 전체 nav와 제품 내부 nav가 공존한다.
4. 투자 화면은 정보 탐색보다 '발견/랭킹/이슈'를 우선한다.
5. 계좌 화면은 데이터보다 즉시 행동을 우선한다.

### 8.12 그대로 따라 하지 않고 가져갈 수 있는 안전한 결론

**가져가도 되는 것**

- 카드 기반 정보 분해
- 큰 타이포그래피
- 빈 상태 설계
- 화면당 대표 CTA
- progressive disclosure
- 개인화된 섹션 제목
- 리스트 행의 아이콘/라벨/상태/chevron 구조
- 데이터 화면의 하단 고정 액션
- 검색 우선 전체 메뉴
- 최근 사용/최근 본 항목
- 숫자 + 해석 + 행동의 조합

**피해야 할 것**

- 토스와 거의 같은 다크 팔레트
- 같은 수준의 둥근 카드와 같은 비례
- 유사한 하단 플로팅 탭바 형태
- 토스식 스파클/아이콘 문법
- "혜택", "맞춤형 서비스", "찾아보기" 같은 문구를 동일한 맥락으로 복제
- 같은 정보 순서와 같은 CTA 배치
- 토스 특유의 계좌/증권/피드 조합을 그대로 재현
- 동일한 광고 삽입 위치와 카드 구조
- 토스 브랜드처럼 보이는 마이크로카피

### 8.13 가계부/자산관리 앱 적용 시 변환

| 토스식 원리 | 가계부/자산관리 앱 적용 |
|---|---|
| 계좌 잔액을 크게 보여줌 | 이번 달 남은 돈/위험 구간을 크게 보여줌 |
| 이자 받기 CTA | 이번 달 정리하기 / 지출 입력하기 / 예산 조정하기 |
| 실시간 차트 | 이번 달 지출 급증 카테고리 |
| 관심 종목 | 관심 예산/자주 보는 카테고리 |
| 발견 탭 | 절약 기회 / 구독 탐지 / 이상 지출 발견 |
| 피드 | 커뮤니티보다는 인사이트/리포트 중심이 안전 |
| 전체 메뉴 | 기능 목록보다 검색 + 최근 사용 중심 |
| 계좌 관리 | 연결 계좌/카드/자동분류 규칙 관리 |

가장 중요한 변환은 이것이다.

> 토스는 "금융 거래"를 "생활 행동"으로 번역한다.
> 네 앱도 "가계부 입력"을 "돈을 이해하고 다음 행동을 정하는 경험"으로 번역해야 한다.

### 8.14 GPT 분석의 10줄 결론

1. 큰 제목과 현재 상태로 화면 맥락을 즉시 제공한다.
2. 카드와 리스트로 복잡한 금융 기능을 작은 행동 단위로 쪼갠다.
3. 한 화면에는 하나의 대표 CTA만 강하게 둔다.
4. 빈 상태도 다음 행동으로 이어지게 만든다.
5. 개인화 문구로 사용자가 앱 안에 존재한다는 느낌을 만든다.
6. 금융 숫자는 숨기지 않지만, 쉬운 언어와 함께 보여준다.
7. 상세 기능은 늦게 보여주는 progressive disclosure를 쓴다.
8. 랭킹, 이슈, 피드로 금융 앱의 사용 빈도를 높인다.
9. 광고/혜택/추천을 서비스 흐름 안에 녹인다.
10. 전체 앱은 슈퍼앱, 개별 화면은 단일 목적 앱처럼 설계한다.

좋아. 웹검색 결과까지 반영하면, 앞선 분석은 대체로 맞지만 **핵심 표현을 조금 바꿔야 한다.**

이전에는 "토스는 복잡한 금융을 카드/리스트/CTA로 쪼개서 쉽게 만든다"고 정리했는데, 공식 아티클들을 보면 더 정확한 표현은 이거다.

> 토스 UI의 본질은 "예쁜 단순함"이 아니라, **인지 비용·심리 비용·노동 비용을 줄이는 운영 원칙의 시각화**다.

토스가 공개한 Product Principles의 중심은 **Simplicity**이고, 토스는 단순함을 "사용자가 제품을 쓰기 위해 특별히 알아야 하거나 배워야 할 것이 없는 상태"로 정의한다. 2025년 토스피드의 Product Principles 글에서는 Casual Concept, Simple Policy, Value First, Clear Action, Context Based, Easy to Answer, Explain Why, One Thing, No More Loading, Minimum Features를 핵심 원칙으로 제시한다. 2023년 토스테크 글도 단순함을 만들기 위해 인지 비용, 심리 비용, 노동 비용을 줄여야 한다고 설명한다. ([Toss][w1])

### 9.1 웹검색 후 보강된 핵심 결론

토스 UI를 분석할 때 가장 중요한 프레임은 "디자인 스타일"이 아니라 **Product Principle → UX Writing → Design System → Research Loop**의 연결이다.

스크린샷에서 보이는 다크모드, 둥근 카드, 큰 글자, 파란 CTA는 표면이다. 더 깊은 층에는 다음 구조가 있다.

| 층 | 토스의 공개 자료에서 확인되는 원칙 | 스크린샷에서 보이는 결과 |
|---|---|---|
| Product Principle | Simplicity, One Thing, Clear Action, Value First, Context Based | 화면마다 대표 행동이 하나씩 강하게 노출됨 |
| UX Writing | Clear, Concise, Casual, Respect, Emotional | 금융 용어를 짧고 친근하게 풀어씀 |
| Design System | 다양한 케이스 패턴화, 심미적 경험, 모든 유저 사용성 | 계좌/설정/증권/피드가 같은 문법으로 보임 |
| Research | UT, 인터뷰, FGI, 설문, 탐색 경험 정량화 | 사용자가 실제로 막히는 지점을 줄이는 방향으로 개선됨 |
| 증권 UX R&D | 투자 진입장벽을 낮추고 복잡함을 덜어냄 | 발견, 관심, AI 이슈, 랭킹, 피드 중심의 증권 경험 |

즉, 토스 UI는 "화면을 잘 그린 결과"라기보다, **사용자가 망설이는 순간을 계속 제거한 결과물**에 가깝다.

### 9.2 Product Principle 관점에서 다시 본 스크린샷

#### 9.2.1 Simplicity는 '미니멀한 화면'이 아니다

토스의 Simplicity는 화면에 요소를 적게 넣는다는 뜻이 아니다. 스크린샷을 보면 정보는 오히려 많다. 지수, 등락률, 이벤트, 광고, 랭킹, 피드, 계좌번호, 잔액, 거래내역, 금리, 카드, 자동이체, 외화까지 한 앱에 다 있다.

그런데 복잡하게 느껴지지 않는 이유는, 토스가 복잡함을 **정책·문구·컴포넌트·행동 단위**로 쪼갰기 때문이다. 공식 글에서도 Product Principle의 핵심은 사용자가 "알아야 할 것"과 "배워야 할 것"을 줄이는 것이며, 이를 위해 Casual Concept, One Thing, Clear CTA, Less Policy, Easy to Answer, Minimum Features 같은 하위 원칙을 둔다고 설명한다. ([토스 기술 블로그, 토스 테크][w2])

| 화면 | 사용자가 배워야 할 것을 줄이는 방식 |
|---|---|
| 토스증권 홈 | "증권 계좌 개설"을 이벤트/퀴즈/CTA로 번역 |
| 관심 탭 | "관심종목 관리"를 "찾아보기"라는 단일 행동으로 번역 |
| 발견 탭 | "시장 탐색"을 자산군 칩, AI 이슈, 실시간 차트로 번역 |
| 계좌 홈 | "은행 계좌 관리"를 잔액, 이자, 보내기, 채우기로 번역 |
| 설정 | 복잡한 설정을 카드 그룹과 행 리스트로 번역 |
| 계좌 상세 | 거래내역 탐색 중에도 보내기/채우기 행동을 고정 |

그래서 토스식 단순함의 정의는 "요소가 적다"가 아니라 **사용자가 다음 행동을 추론하지 않아도 된다**에 가깝다.

#### 9.2.2 One Thing: 한 화면은 하나의 핵심 메시지를 가진다

토스 공식 Product Principles에는 **One Thing**이 있다. 각 페이지는 하나의 명확한 핵심 메시지를 전달해야 한다는 원칙이다. ([Toss][w1])

| 화면 | 화면의 One Thing |
|---|---|
| 토스증권 홈 | 증권 계좌를 만들거나 증권 서비스를 시작하라 |
| 관심 | 궁금한 종목을 찾아 관심 루틴을 만들라 |
| 발견 | 지금 시장에서 볼 만한 이슈/종목을 발견하라 |
| 피드 | 시장 관련 콘텐츠와 커뮤니티 반응을 읽어라 |
| 전체 메뉴 | 원하는 서비스를 검색하거나 최근 서비스를 다시 실행하라 |
| 계좌 홈 | 현재 잔액을 확인하고 보내기/채우기를 하라 |
| 계좌 상세 | 거래내역을 확인하고 필요하면 바로 돈을 이동하라 |
| 계좌 관리 | 계좌 속성과 이체 조건을 관리하라 |

이 때문에 화면마다 정보는 많지만, 주인공은 하나다. 이게 "토스 느낌"의 핵심이다.

#### 9.2.3 Clear Action: CTA가 애매하지 않다

토스의 Product Principles 중 **Clear Action**은 사용자가 다음에 무엇을 해야 하는지 한눈에 알 수 있어야 한다는 원칙이다. 공식 글에서는 "글을 읽지 않아도 사용자가 다음 행동을 알아야 한다"는 취지로 설명한다. ([Toss][w1])

스크린샷에서 CTA는 거의 항상 명확하다: 문제 보기, 토스증권 계좌 만들기, 찾아보기, 간편 홈 보기, 이자 받기, 채우기, 보내기, 복사, 보기.

흥미로운 점은 CTA 문구가 "기능명"이 아니라 "행동명"이라는 것이다. "계좌개설 신청 페이지 이동"이 아니라 "계좌 만들기", "거래내역 조회"가 아니라 "보기", "입금"이 아니라 "채우기"다. 사용자는 금융 기능명을 이해하지 않아도 행동을 이해한다.

#### 9.2.4 Easy to Answer: 사용자가 어려운 결정을 하지 않게 한다

토스의 **Easy to Answer** 원칙은 특히 중요하다. 토스테크의 보험 청구 사례에서 "3초 안에 답이 안 나오면 어려운 질문"이라고 설명하고, "전문가 도움받을래 직접 할래?"보다 "서류 있어?"처럼 사용자가 쉽게 대답할 수 있는 질문으로 바꿔 이탈을 줄였다고 설명한다. ([토스 기술 블로그, 토스 테크][w3])

| 복잡한 질문 | 토스가 화면에서 바꾼 질문 |
|---|---|
| 어떤 금융 서비스를 이용하시겠습니까? | 최근 서비스 / 바로가기 / 검색 |
| 어떤 증권 정보를 보고 싶습니까? | 거래대금, 거래량, 급상승, 급하락, 인기 |
| 어떤 종목을 관리하시겠습니까? | 최근 본 종목이 없어요 → 찾아보기 |
| 계좌에서 어떤 작업을 하시겠습니까? | 채우기 / 보내기 |
| 어떤 설정을 바꾸시겠습니까? | 언어, 알림, 화면 테마, 인증서, 비밀번호 |

토스는 사용자를 "스스로 분류하고 결정하는 사람"으로 두지 않는다. **앱이 먼저 가능한 선택지를 쉬운 질문 형태로 제시한다.**

#### 9.2.5 Value First: 비용보다 가치를 먼저 보여준다

토스의 **Value First, Cost Later** 원칙은 "비용을 말하기 전에 가치를 명확히 전달해야 한다"는 것이다. 토스테크의 '내 보험' 사례에서는 긴 인증 과정을 줄일 수 없었기 때문에, 사용자가 그 과정을 시작하기 전에 어떤 가치를 얻게 되는지 먼저 보여주는 방식으로 개선했다고 설명한다. ([토스 기술 블로그, 토스 테크][w4])

| 기능 중심 표현 | 토스식 가치 중심 표현 |
|---|---|
| 증권 계좌 개설 | 1만원 받는 OX 퀴즈 |
| 이자 조회 | 이자 받기 |
| 보험 조회 | 내 보험을 한 번에 확인 |
| 자동이체 관리 | 자동이체 8개 |
| 외화 잔액 | 미국 달러 1.00 USD, 전체 외화 보기 |
| 추천 서비스 | 오늘 받을 수 있는 혜택 |

그래서 광고/혜택/추천이 많아도 노골적 배너처럼 덜 느껴진다.

#### 9.2.6 Context Based: 직전 행동과 다음 행동이 자연스럽게 이어진다

토스의 Product Principles 중 **Context Based**는 사용자의 여정 전체가 자연스럽게 이어져야 한다는 원칙이다. ([Toss][w1])

스크린샷에서 이 원칙은 특히 계좌 화면과 증권 화면에서 보인다. 계좌 홈에서는 잔액 → 이자 → 최근 거래 → 채우기/보내기 → 자동이체 → 혜택 → 용돈 → 외화로 이어진다. 증권 발견 화면에서는 시장 지수 → 자산군 → 예정 이벤트 → AI 이슈 → 실시간 차트 → 종목 리스트로 이어진다. 즉, 토스의 화면 순서는 단순한 중요도 순서가 아니라 **사용자의 머릿속 질문 순서**에 가깝다.

### 9.3 UX Writing 자료로 보강한 분석

토스 공식 UX Writing 글에 따르면 토스의 라이팅 코어밸류는 Clear, Concise, Casual, Respect, Emotional이다. 특히 Clear는 사용자가 이해할 수 있는 문장을 쓰는 것, Concise는 필요한 정보만 필요한 만큼 전달하는 것, Casual은 금융/IT 업계의 어려운 용어와 딱딱한 뉘앙스를 쉽게 고쳐 쓰는 것, Respect는 혜택을 과장하거나 비용을 숨기지 않는 커뮤니케이션을 뜻한다. ([토스 기술 블로그, 토스 테크][w5])

> 참고: 본 문서 7.2 섹션의 Codex 리뷰 반영 결과와 비교해보면, 9.3의 "5 코어밸류" 묘사가 7.2에서 정정한 분류(코어밸류 = Clear/Concise/Casual + 비공개 2개 / 별도 8가지 라이팅 원칙)와 일부 어긋난다. 외부 자료마다 명명이 흔들리는 영역이므로 정확한 권위는 토스 공식 원문(`toss.tech/article/8-writing-principles-of-toss`)을 1차 출처로 본다.

| 문구 | 작동 방식 |
|---|---|
| "문제 보기" | 이벤트 참여를 시험/퀴즈처럼 가볍게 만듦 |
| "찾아보기" | 빈 상태를 부정적 실패가 아니라 다음 행동으로 바꿈 |
| "이자 받기" | 금융 개념을 즉시 보상처럼 표현 |
| "채우기" | 입금보다 생활 언어에 가까움 |
| "보내기" | 송금보다 행동성이 강함 |
| "오늘 받을 수 있는 혜택" | 사용자가 지금 얻을 것을 먼저 제시 |
| "최근 본 종목이 없어요" | 탓하지 않고 상태만 설명 |

토스의 라이팅 원칙 중에는 Predictable hint, Weed cutting, Remove empty sentences, Focus on key message, Easy to speak, Suggest over force, Universal words, Find hidden emotion도 있다. ([토스 기술 블로그, 토스 테크][w5])

### 9.4 디자인 시스템 자료로 보강한 분석

토스는 TDS를 "메이커의 생산성과 사용자의 일관된 경험"을 책임지는 도구로 운영한다고 설명한다. 메뉴 컴포넌트 설계 사례에서는 다양한 케이스를 패턴화하고, 심미적 경험을 설계하고, 모든 유저의 사용성을 지키는 세 가지를 디자인 개선 요소로 제시한다. ([토스 기술 블로그, 토스 테크][w6])

| 반복 컴포넌트 | 등장 위치 | 역할 |
|---|---|---|
| 큰 화면 제목 + 보조 지표 | 증권, 관심, 발견, 피드 | 현재 위치와 시장 상태를 동시에 제공 |
| Hero Card | 증권 홈, 계좌 홈 | 가장 중요한 상태/전환을 크게 표현 |
| Grouped List | 전체 메뉴, 설정, 계좌 관리 | 복잡한 기능을 소화 가능한 묶음으로 분리 |
| Row + right value + chevron | 설정, 계좌 관리, 전체 메뉴 | 상태 표시와 이동 가능성을 동시에 표현 |
| Ranking Row | 발견, 실시간 차트 | 실시간 데이터를 스캔 가능한 리스트로 변환 |
| Empty State | 관심 | 비어 있는 상태를 온보딩으로 전환 |
| Sticky Action | 계좌 상세 | 데이터를 보다가 바로 행동하게 함 |
| Floating Bottom Nav | 증권 내부 화면 | 제품 내부의 독립적인 이동 체계 제공 |

### 9.5 리서치 자료로 보강한 분석

토스는 데이터 기반 의사결정뿐 아니라 사용자 목소리를 중요하게 보고, UT, 심층 인터뷰, FGI, 설문조사 등을 통해 사용자 경험을 조사한다고 설명한다. 또 2025년 TOSS MAKERS CONFERENCE 세션 소개에서는 사용자가 원하는 서비스나 기능을 잘 찾는지 확인하기 위해 정성 리서치로만 파악하던 탐색 경험을 EVR과 TNS라는 도구로 정량화하려 했다고 설명한다. ([Toss][w7])

| 관찰되는 UI | 숨어 있는 리서치 가설 |
|---|---|
| 바로가기 / 최근 서비스 | 사용자는 전체 기능 목록보다 최근 행동을 재개하고 싶어 한다 |
| 검색창이 전체 메뉴 최상단 | 슈퍼앱은 탐색보다 검색이 더 빠를 수 있다 |
| 관심 탭 empty state | 빈 상태를 방치하면 사용자는 다음 행동을 모른다 |
| 발견 탭 랭킹 | 초보 투자자는 복잡한 차트보다 "지금 움직이는 것"을 먼저 본다 |
| AI 이슈 | 사용자는 원문 뉴스보다 해석 가능한 요약을 원한다 |
| 계좌 상세의 하단 고정 CTA | 거래내역 확인 중에도 송금/입금으로 바로 전환하고 싶어 한다 |
| 설정 카드 그룹 | 많은 설정을 한 목록에 두면 탐색 피로가 커진다 |

UI 패턴을 복제하는 게 아니라 **네 앱에서도 사용자가 실제로 막히는 지점을 찾아야 한다**.

### 9.6 토스증권 자료로 보강한 분석

토스증권 UX R&D Team은 "왜 아직도 많은 사람들이 투자를 어렵게 느낄까?"라는 질문에서 출발해, 복잡함을 덜어내고 진입장벽을 낮추는 UX를 실험한다고 설명한다. ([Toss][w8])

전통 증권 앱의 중심은 주문/호가/차트/계좌/보유 종목/조건 검색/리서치 리포트인 반면, 토스증권 스크린샷의 중심은 관심/발견/피드/AI 시그널/실시간 이슈/랭킹/이벤트/커뮤니티다. 즉, 토스증권은 "주문 앱"이라기보다 **투자 정보 소비와 루틴 형성 앱**에 가깝다.

토스 UX 리서처 글에서는 초보 투자자가 관심종목 등록이나 매수 타이밍 설정에서 어려움을 겪고, 숙련된 투자자는 자기만의 루틴이 있다는 점을 발견했다고 설명한다. 이후 지정가 알림, 조건주문처럼 루틴을 돕는 기능이 등장했다고 설명한다. ([토스 기술 블로그, 토스 테크][w9])

| UI | 루틴 형성에서의 역할 |
|---|---|
| 관심 탭 | 종목을 계속 지켜보는 습관 만들기 |
| 최근 본 | 탐색했던 종목으로 돌아오기 |
| 하트 | 관심 루틴에 편입하기 |
| 발견 랭킹 | 오늘 볼 후보군 만들기 |
| AI 시그널 | 시장 해석 루틴 만들기 |
| 피드 | 시장 감정/커뮤니티 반응 확인 루틴 만들기 |

### 9.7 AI 이슈와 시장 정보 UI의 의미

토스증권의 증시 캘린더 관련 리서치 글에서는 사용자가 유튜브, 커뮤니티 등 여러 채널을 돌아다니며 주요 증시 일정을 찾고 있었다고 설명한다. 그래서 단순 일정 표시를 넘어 투자 흐름을 정리해주는 도구로 고도화했고, AI 주간 요약이나 지표 해석 기능 같은 장치를 제공한다고 설명한다. ([토스 기술 블로그, 토스 테크][w10])

스크린샷의 "AI 실시간 이슈", "토스증권 AI 시그널"이 해결하려는 문제:

> 시장 정보는 너무 많고 흩어져 있으며, 초보 투자자는 무엇이 중요한지 판단하기 어렵다.

즉 토스증권의 AI UI는 "AI 기능"이라기보다 **정보 과잉을 줄이는 인터페이스 장치**다. 하지만 AI가 요약하고 우선순위를 정하면, 사용자는 그것을 "시장 해석"이 아니라 "투자 판단 신호"로 받아들일 수 있다. 따라서 출처, 요약 기준, 지연 시간, 불확실성, 투자 권유 아님 고지가 필요하다.

### 9.8 외부 비판 자료까지 반영한 위험 분석

웹검색 후 가장 중요하게 추가해야 할 부분은 이것이다.

> 토스식 "쉬운 금융"은 저위험·생활금융에서는 강력하지만, 고위험 투자상품에서는 위험을 희석할 수 있다.

2025년 여러 보도에서 토스증권의 미수거래, 해외옵션 UI를 둘러싼 논란이 다뤄졌다. 대한금융신문은 미수거래의 용어가 바뀌었더라도 지나치게 간편한 화면 구성 때문에 투자자의 경각심이 낮아진다는 비판이 나왔다고 보도했다. ([DFT 대한금융신문][w11])

해외옵션과 관련해서도 매경이코노미는 복잡한 옵션 구조를 퀴즈형 인터페이스로 단순화하고 높은 수익 가능성을 전면에 배치했다는 비판을 보도했다. ([매일경제][w12])

토스의 원칙 중 **Casual Concept**은 어려운 개념을 친숙하게 바꾸는 것이다. 그런데 고위험 상품에서는 친숙한 표현이 위험을 낮게 인식시키는 부작용을 만들 수 있다.

| 기존 결론 | 웹검색 후 수정된 결론 |
|---|---|
| 금융을 쉽게 만드는 것이 좋다 | 저위험 금융은 쉽게, 고위험 금융은 일부러 무겁게 만들어야 한다 |
| CTA는 명확할수록 좋다 | 위험 행동의 CTA는 명확하되, 위험 인식도 함께 명확해야 한다 |
| Casual Concept은 좋은 원칙이다 | Casual Concept은 위험의 본질을 희석하지 않을 때만 좋다 |
| No More Loading은 좋다 | 고위험 상품은 의도적 마찰이 필요할 수 있다 |
| One Thing은 좋다 | 고위험 화면의 One Thing은 "거래"가 아니라 "위험 이해"여야 할 수 있다 |

즉, 토스식 UX의 한계는 **너무 잘 만든 UX가 사용자를 너무 쉽게 움직이게 만들 수 있다**는 점이다.

### 9.9 스크린샷별 추가 재해석

- **증권 홈**: 기존 분석의 "온보딩 + 이벤트 + 계좌개설 전환"에 **Value First**를 더 강하게 적용. "퀴즈 풀고 1만원 받으세요"가 비용·인증·계좌 개설 부담보다 보상을 먼저 보여주는 구조.
- **관심 탭**: One Thing + Clear Action + Easy to Answer 결합. 사용자를 탓하지 않고 다음 행동 하나만 제시.
- **발견 탭**: 단순한 콘텐츠 허브가 아니라 **투자 루틴의 시작점**. 자산군 칩, 이벤트, AI 이슈, 랭킹을 통해 "오늘 볼 후보"를 만든다. FOMO 위험도 큼.
- **피드**: UX적으로 강력하지만 금융 서비스로서는 가장 위험. 권위 배지로 과신, 커뮤니티 글이 투자 조언으로 소비, 극단적 수익/손실 서사가 FOMO 유발. 가계부/자산관리 앱이라면 커뮤니티보다 "내 데이터 기반 인사이트 피드"가 안전.
- **계좌 홈**: 가장 토스다운 화면. 계좌 상세정보가 아니라 **계좌를 중심으로 한 생활 행동 패널**.

### 9.10 네 앱에 적용할 때의 수정된 설계 원칙

**가져갈 것**

1. **One Screen, One Job** — 한 화면은 하나의 과업만.
2. **Value First** — "왜 이걸 해야 하는지" 먼저 보여줌.
3. **Easy to Answer** — 어려운 선택을 시키지 않음. 선택지는 2~4개 이하.
4. **Clear Action** — CTA는 기능명이 아니라 행동명.
5. **Context Based** — 사용자 흐름을 화면 순서에 반영.
6. **Design System화** — 카드/행/CTA를 반복하되 토스 스타일 복제 금지. 필수 컴포넌트: `StatusHero`, `InsightCard`, `ActionRow`, `GroupedList`, `TransactionRow`, `EmptyState`, `StickyActionBar`, `CategoryChip`, `MonthlySummary`, `RiskNotice`.

**가져가면 위험한 것**

1. **고위험 행동까지 너무 쉽게 만드는 것** — 미수거래·옵션 UX 논란 ([DFT 대한금융신문][w11])이 보여주듯 의도적 마찰이 필요.
2. **광고를 기능처럼 보이게 만드는 것** — 광고/추천/제휴/혜택은 명확히 라벨링.
3. **피드/커뮤니티로 체류시간을 늘리는 것** — 의사결정 왜곡 위험.
4. **같은 다크모드·둥근 카드·파란 CTA 조합** — 시각적 언어는 다르게.

### 9.11 기존 정합성 검증의 업데이트

스크린샷에서 관찰한 패턴이 토스의 공식 Product Principles와 거의 1:1로 맞는다.

| 스크린샷에서 관찰한 패턴 | 토스 공식/분석 자료의 대응 원칙 |
|---|---|
| 큰 제목 + 현재 상태 | One Thing, Clear Action |
| 계좌개설 이벤트 선노출 | Value First |
| 쉬운 CTA 문구 | Clear Action, Easy to Speak |
| 빈 상태 + 찾아보기 | Easy to Answer, One Thing |
| 발견 탭 랭킹/AI 이슈 | Context Based, Casual Concept |
| 계좌의 보내기/채우기 | Low-cost Action, Clear Action |
| 설정의 그룹 카드 | Simple Policy, Design System |
| 최근 서비스/개인화 | Personalize, Context Based |
| 광고처럼 보이지 않는 혜택 | No Ads Patterns, Value First |
| 증권 피드/랭킹/관심 | 투자 루틴 형성, 정보 탐색 부담 감소 |

다만 새로운 조건을 추가해야 한다.

> 구조적으로 재구성은 가능하지만, 고위험 금융 행동까지 동일한 매끄러움으로 재구성하면 UX 윤리 문제가 생길 수 있다.

### 9.12 최종 보정된 결론 (GPT 12원칙)

1. **Simplicity는 시각적 미니멀리즘이 아니라 비용 제거다.**
2. **복잡한 금융 개념을 Casual Concept으로 생활 언어화한다.**
3. **한 화면은 하나의 핵심 메시지만 갖는다.**
4. **CTA는 기능명이 아니라 사용자의 행동명으로 쓴다.**
5. **사용자에게 어려운 질문을 던지지 않는다.**
6. **가치를 먼저 보여주고, 비용·입력·인증은 나중에 요구한다.**
7. **사용자의 직전 맥락에서 다음 화면이 자연스럽게 이어진다.**
8. **디자인 시스템으로 카드/행/CTA/그룹/리스트 문법을 반복한다.**
9. **UX Writing이 UI의 절반 이상을 차지한다.**
10. **리서치와 지표로 사용자가 막히는 지점을 계속 줄인다.**
11. **증권 UX에서는 정보 과잉을 랭킹·AI 요약·루틴으로 재구성한다.**
12. **하지만 고위험 금융상품에서는 '쉽게 만드는 것'이 오히려 위험할 수 있다.**

가장 중요한 결론:

> 사용자가 지금 무엇을 알고 싶고, 무엇을 결정하기 어려워하고, 어떤 행동으로 넘어가야 하는지를 화면 하나하나에서 명확히 번역하는 능력.

그리고 토스와 다르게 해야 할 것:

> 위험하거나 되돌리기 어려운 행동은 절대 지나치게 매끄럽게 만들지 말고, 사용자가 위험을 충분히 인식하도록 의도적인 마찰을 설계하는 것.

## 10. Claude(1~7장) vs GPT-5(8~9장) 종합 비교

두 모델이 동일 스크린샷으로 독립 분석한 결과를 한 표에 정렬:

| 비교 축 | Claude (1~7장) | GPT-5 (8~9장) |
|---|---|---|
| 추출 정책 수 | 12개 (P1~P12), 외부 검증 후 +4개 (P0/P13~P15) | 5개 핵심 → 12개 최종 보정 |
| 강조점 | **디자인 토큰·컴포넌트 카탈로그** (시스템 추상화) | **사용자 과업·행동 흐름** (UX 의도) |
| 외부 자료 활용 | 7장에서 TDS·라이팅 8원칙·Simplicity 컨퍼런스 인용 | 9장에서 Product Principles·UX Writing·외부 비판 보도까지 폭넓게 인용 |
| 위험·윤리 시각 | 약함 (P7 광고 라벨링 정도) | 강함 (FOMO·고위험 상품 UX·미수거래 논란까지 다룸) |
| 재구성 시뮬레이션 | 10/10 화면 골조 재구성 검증 | 동일하게 10화면 재구성 가능 확인 |
| 새로 발견한 핵심 원칙 | **P0 한 화면 한 기능** / 토큰 자동화 / 컴포넌트 거버넌스 | **Value First** / **Easy to Answer** / **Context Based** / 고위험 영역의 의도적 마찰 |

**두 분석이 합쳐서 보여주는 것**

- 같은 결론에 도달한 부분(빈 상태가 기능이다, 한 화면 한 행동, 카드 그룹, 인라인 액션, 개인화의 IA화)이 매우 많아 **추출의 재현성이 높다**.
- Claude는 토스의 **시각·시스템 레이어**를 잘 풀었고, GPT-5는 토스의 **제품 원칙·UX 윤리 레이어**를 잘 풀었다. 둘은 보완 관계.
- 가장 중요한 보강: **"잘 만든 UX가 사용자를 너무 쉽게 움직이게 만들 수 있다"** 라는 위험 인식. 이건 Claude 분석에는 거의 없던 축이며, 가계부/자산관리 같은 저위험 도메인에는 덜 적용되지만 *되돌리기 어려운 행동*이 포함된다면 반드시 고려해야 함.

## 11. 종합 보고 — xef-scale을 위한 디자인 시스템 제안

> 1~10장의 모든 추출 결과(Claude 16개 정책 P0~P15 + GPT 12원칙 + 외부 비판 보강)를 통합해서 **xef-scale에 적용 가능한 디자인 시스템 제안서**로 정리한다. 본 장은 디스커버리 노트의 일부이므로 즉시 권위가 아니며, 채택 시 `docs/decisions/`에 ADR로 승격되어야 한다 (`docs/discovery/README.md` 정책).
>
> **전제**: xef-scale = Rails 8 + Tailwind CSS 4 + Hotwire(Turbo/Stimulus) + 모바일 퍼스트 + 워크스페이스 멀티테넌트. 토스 시각 표현(다크 팔레트·둥근 카드·파란 CTA·플로팅 탭바 곡률)은 **복제하지 않는다**. 가져가는 것은 *원리*다.

### 11.1 도메인 번역 — 토스 원칙을 xef-scale 어휘로

xef-scale의 핵심 객체와 행동은 토스와 다르다. 원리를 그대로 가져오되 어휘는 **지출 추적 + 가족 공유 + AI 파싱 + 카테고리화 + 검토/커밋**의 컨텍스트로 번역한다.

| 토스 원리 | xef-scale 번역 |
|---|---|
| P0 한 화면 한 기능 | 한 화면 한 과업: 입력 / 검토 / 분류 / 회고 / 공유 중 하나만 |
| P1 숫자가 주인공 | 잔액 대신 **이번 달 지출 합계 · 카테고리 합계 · 예산 대비**가 hero |
| P2 라이브 컨텍스트 | 화면 헤더 옆 인라인 컨텍스트는 시장 지수가 아니라 **이번 달 vs 지난달 델타 / 예산 잔여 / 검토 대기 건수** |
| P3 개인화는 골조 | 워크스페이스 이름·멤버 닉네임을 헤더·빈상태·섹션 헤더에 항상 삽입 |
| P4 Dark-first | xef-scale은 light-first가 합리적 (가계부는 일상 도구 + 영수증/명세서가 라이트 톤). 그러나 OS 다크 모드는 **1:1 페어 토큰**으로 자동 대응 |
| P5 하나의 액션 컬러 | xef-scale 브랜드 컬러 1개만 1차 CTA·링크·활성 탭에 사용. **상승/하락·증감과 절대 같은 톤을 쓰지 않는다** |
| P6 AI 별도 채널 | Gemini가 만든 카테고리 제안·중복 후보·금액 추출은 **AI 마커(✨ 또는 별도 보더)** 로 구분 |
| P7 광고 라벨링 정직 | xef-scale에 광고는 없음. 단 Phase C 광고 도입 시 무조건 "AD" 명시 (현 시점 비대상) |
| P8 인라인 액션 | 거래 목록 row에서 카테고리 즉시 변경 / 중복 표시 / 코멘트 추가 / 용돈 마킹 가능해야 함 |
| P9 빈 상태는 기능 | 거래 0건·검토 0건·카테고리 0건 빈 상태가 다음 행동(붙여넣기·업로드·직접 입력)으로 이어져야 함 |
| P10 묶기(grouping) | 설정·워크스페이스 관리·카테고리 매핑은 카드 그룹으로. 거래는 날짜 그룹 |
| P11 스크롤이 상단을 변환 | 거래 목록 스크롤 시 헤더가 **이번 달 합계 → 보고 있는 날짜 그룹 합계**로 전환 |
| P12 한국 금융 관행 | `원` 후치 / 천단위 콤마 / 소수 없음 / "님" 호칭 / 빨강=수입·환급, 파랑=지출 (서비스 컨벤션 명시 필요) |
| P13 라이팅도 시스템 | 마이크로카피 키 테이블화. 같은 의미는 같은 표현 (예: "기록하기" vs "추가하기" 혼용 금지) |
| P14 토큰 자동화 | Tailwind 4의 `@theme` + CSS 변수로 토큰화. 컴포넌트 partial이 토큰만 참조 |
| P15 컴포넌트 거버넌스 | partial 신설 시 PR description에 "왜 기존 partial로 못 푸는가" 1줄 필수 |
| GPT One Thing | 위 P0과 동일 |
| GPT Value First | 인증·복잡한 입력 전에 "이걸 하면 무엇이 보이는지" 먼저 보여줌 |
| GPT Easy to Answer | "이 거래 카테고리는?" 같은 미확정 질문은 2~4개 후보 + AI 추천 1개로 제시 |
| GPT Clear Action | CTA는 "Transaction#create" 같은 기능명 금지. "기록하기 / 반영하기 / 보내기 / 정리하기" 같은 행동명 |
| GPT Context Based | 검토 화면 → 중복 해결 → 커밋 → 카테고리 학습 결과 확인이 자연스럽게 이어져야 함 |
| **GPT 윤리 보강** | 금융 거래 수정·삭제·일괄 변경처럼 **되돌리기 어려운 행동**은 의도적 마찰 (확인 / 영향 범위 / 멤버 알림) 추가 |

### 11.2 디자인 토큰

Tailwind CSS 4의 `@theme` 블록과 CSS 변수로 정의한다. **모든 색·간격·라운드 값은 토큰으로만 사용**하며 컴포넌트는 직접 HEX를 쓰지 않는다.

#### 11.2.1 컬러 토큰 (시맨틱 우선)

```css
@theme {
  /* Surface — light & dark 자동 페어 */
  --color-bg-base:        light-dark(#fafafa, #0f1014);
  --color-bg-surface:     light-dark(#ffffff, #181a1f);
  --color-bg-elev:        light-dark(#f4f4f5, #21242b);
  --color-bg-overlay:     light-dark(#00000080, #00000099);

  /* Text */
  --color-text-primary:   light-dark(#0a0a0a, #fafafa);
  --color-text-secondary: light-dark(#525252, #a3a3a3);
  --color-text-tertiary:  light-dark(#a3a3a3, #6b7280);
  --color-text-inverse:   light-dark(#ffffff, #0a0a0a);

  /* Border / Divider */
  --color-border-subtle:  light-dark(#e5e5e5, #2a2d33);
  --color-border-strong:  light-dark(#d4d4d4, #3f434b);

  /* Brand action — xef-scale 고유 컬러 (예: 차분한 인디고 계열, 토스 블루와 구분) */
  --color-action:         light-dark(#4f46e5, #818cf8);
  --color-action-hover:   light-dark(#4338ca, #6366f1);
  --color-action-on:      #ffffff;

  /* Semantic — 금액 증감 (한국 관행: 지출=blue tone, 수입/환급=red tone) */
  /* 단, 브랜드 action과 톤이 겹치지 않도록 일부러 회색이 섞인 톤으로 분리 */
  --color-spend:          light-dark(#1e40af, #60a5fa);   /* 지출 */
  --color-income:         light-dark(#b91c1c, #f87171);   /* 수입·환급 */

  /* AI — 보라 계열로 격리 */
  --color-ai-accent:      light-dark(#7c3aed, #a78bfa);
  --color-ai-surface:     light-dark(#f5f3ff, #1e1b2e);
  --color-ai-border:      light-dark(#ddd6fe, #5b21b6);

  /* Status */
  --color-warning:        light-dark(#b45309, #f59e0b);
  --color-error:          light-dark(#b91c1c, #f87171);
  --color-success:        light-dark(#15803d, #4ade80);

  /* Pending review · duplicate */
  --color-pending:        light-dark(#a16207, #fbbf24);
  --color-duplicate:      light-dark(#9a3412, #fb923c);
}
```

규칙:
- **하나의 브랜드 액션 컬러**가 1차 CTA, 활성 탭, 링크에 사용된다.
- **지출/수입 컬러는 액션 컬러와 톤 분리**(채도·명도 차이로 구분). 행동을 부추기지 않기 위한 의도적 분리.
- **AI 출력물은 보라 계열로 격리**. 절대 브랜드 액션 컬러로 표시하지 않는다 (P6 + 윤리 보강).
- **pending/duplicate는 별도 색**: 검토 대기·중복 의심처럼 "사용자 결정이 필요한 상태"는 액션 컬러로 칠하지 않고 amber/orange로 분리.

#### 11.2.2 타이포 토큰

```css
@theme {
  --font-sans: "Pretendard Variable", "Pretendard", system-ui, sans-serif;
  --font-mono: ui-monospace, "SF Mono", monospace;  /* 금액 정렬용 tabular-nums */

  --text-hero: 2.5rem;     /* 이번 달 합계 등 */
  --text-title: 1.5rem;
  --text-section: 1.125rem;
  --text-body: 1rem;
  --text-meta: 0.875rem;
  --text-caption: 0.75rem;
  --text-disclaimer: 0.6875rem;
}
```

- 금액(`Transaction#amount`, 카테고리 합계, 예산)은 **항상 tabular-nums + 천단위 콤마**. 폰트 변동에도 자리수 정렬 유지.
- hero는 **이번 달 지출 합계 또는 보고 있는 화면의 1번 숫자**에만 사용.

#### 11.2.3 간격·라운드·그림자

```css
@theme {
  --spacing-row:    0.75rem;
  --spacing-card:   1.25rem;
  --spacing-section: 2rem;

  --radius-chip:    9999px;
  --radius-card:    1rem;
  --radius-group:   1.5rem;
  --radius-input:   0.75rem;

  --shadow-card:  0 1px 2px rgba(0,0,0,0.04);
  --shadow-elev:  0 4px 16px rgba(0,0,0,0.08);
  --shadow-sheet: 0 -2px 24px rgba(0,0,0,0.12);  /* 바텀시트 */
}
```

- 토스의 둥근 정도(매우 큰 라운드)를 그대로 쓰지 않는다 (시각 모방 회피, P 11.0).
- 그림자는 최소화: 라이트 모드 카드 1단계, 다크 모드는 그림자 대신 surface 단계 차이로 elevation 표현.

#### 11.2.4 모션 토큰

```css
@theme {
  --duration-instant: 100ms;
  --duration-fast:    150ms;
  --duration-base:    220ms;
  --duration-slow:    400ms;
  --ease-standard:    cubic-bezier(0.2, 0, 0, 1);
  --ease-emphasized:  cubic-bezier(0.3, 0, 0.1, 1);
}
```

- 금융 데이터 변경은 **즉시(< 100ms) 확인 피드백** 우선. 화려한 트랜지션은 회피.

### 11.4 페이지 패턴 — xef-scale 주요 화면 골조

각 화면은 위 컴포넌트 조합으로만 구성된다. 신규 시각 자산을 만들지 않는다.

#### 11.4.1 대시보드 (월간)

```
ScreenShell
├── ContextHeader  ("이번 달 지출" + 라이브: 지난달 대비 ±%)
├── HeroStat       (이번 달 합계 + 보조: 예산 잔여 · CTA "예산 조정")
├── SegmentedTabs  (월간 / 주간 / 카테고리 / 캘린더)
├── GroupedListCard "카테고리별"
│   └── Row × N: 카테고리 chip · 합계 · 막대 · chevron
├── GroupedListCard "최근 거래"
│   └── TransactionRow × 5  +  ActionRow("전체 보기")
└── BottomNav
```

대표 행동: **이번 달 어디에 썼는지 파악 → 카테고리/거래로 드릴다운** (One Thing).

#### 11.4.2 거래 목록

```

├── ContextHeader  ("거래" + 라이브: 보고 있는 기간 합계)
├── ChipScroller   (카테고리 / 금융기관 / 상태(committed/pending_review))
├── 날짜 그룹 헤더
│   └── TransactionRow × N
│       ↑ row 내부 인라인 액션: 카테고리 즉시 변경 · 용돈 마킹 · 코멘트 ✨ · 중복 표시
├── (스크롤 시 ContextHeader가 보고 있는 날짜 그룹 합계로 전환 — P11)

```

#### 11.4.3 검토 화면 (reviews/show)

```
ScreenShell
├── ContextHeader  ("검토" + 라이브: "거래 N건 · 중복 의심 M건")
├── HeroStat       (이번 세션의 합계 + AIBadge "Gemini Flash로 파싱됨")
├── AIBadgeCard    "AI가 추정한 카테고리"
│   └── 각 거래에 후보 1+대안 2~3 (Easy to Answer)
├── PendingBadge   (중복 의심 N건이 있으면 노출 + RiskNotice "중복 해결 전에는 반영할 수 없어요")

    └── primary  "거래 내역 반영"  (← Clear Action: 기능명 X, 행동명)
```

### 11.5 UX Writing 가이드

마이크로카피는 디자인 시스템의 일부 (P13). 키 = 의미 / 값 = 화면 표현으로 1:1 관리.

| 키 | 표현 | 사용 컨텍스트 |
|---|---|---|
| `cta.record` | 기록하기 | 거래 신규 |
| `cta.commit_review` | 거래 내역 반영 | 검토 후 커밋 |
| `cta.review_now` | 지금 검토 | 검토 대기 진입 |
| `cta.paste_sms` | 문자 붙여넣기 | 텍스트 파싱 |
| `cta.upload_shot` | 스크린샷 올리기 | 이미지 파싱 |
| `cta.add_manual` | 직접 입력하기 | 수동 입력 |
| `cta.cancel_review` | 검토 취소 | 검토 화면 |
| `empty.no_tx` | 아직 기록된 거래가 없어요 | 거래 0건 |
| `empty.no_review` | 검토할 거래가 없어요 | 검토 0건 |
| `empty.no_category` | 첫 카테고리를 만들어 볼까요? | 카테고리 0건 |
| `ai.suggested` | AI가 추천했어요 | Gemini 출력 라벨 |
| `ai.confidence` | 추천이 확실하지 않아요 — 한 번 더 확인해 주세요 | Gemini 낮은 confidence |
| `pending.dup` | 중복일 수 있는 거래가 있어요 | DuplicateConfirmation pending |
| `risk.commit_locked` | 중복 N건을 먼저 정리해야 반영할 수 있어요 | 검토 commit 비활성 사유 |
| `risk.workspace_leave` | 떠나면 이 워크스페이스의 거래를 더 이상 볼 수 없어요 | 워크스페이스 탈퇴 |
| `risk.bulk_delete` | 거래 N건을 삭제할게요. 되돌릴 수 없어요. | 일괄 삭제 |
| `learn.category` | 다음 같은 가맹점부터는 이 카테고리로 자동 분류돼요 | CategoryMapping 학습 |

규칙:
- **기능명 금지**: "트랜잭션 생성" / "리뷰 커밋" / "카테고리 매핑 저장" 같은 시스템 어휘는 사용자 향 카피에서 금지. (Clear Action)
- **존댓말 통일**: 모든 사용자 향 문장은 "-요/-어요" 종결. (Casual + Respect)
- **위험 행동은 명시적**: "삭제" / "떠나기" 같은 단어 앞에 영향 범위·되돌릴 수 없음을 항상 표시. (윤리 보강)
- **AI 출력은 표지**: "AI가 추천했어요" / "추측이에요" / "한 번 더 확인해 주세요"를 일관되게 사용. 절대 단정 어조 사용 금지.

### 11.6 안전장치 — 의도적 마찰을 두는 곳

토스의 매끄러움이 가계부에서도 항상 옳은 것은 아니다 (9.8). xef-scale에서 의도적으로 마찰을 두는 행동 목록:

| 행동 | 마찰 장치 |
|---|---|
| `Transaction#commit` (검토 → 반영) | 미해결 중복 자동 카운트 → CTA 비활성 + 사유 명시 |
| 거래 일괄 삭제 | RiskNotice + 삭제 건수·합계 표시 + 워크스페이스 다른 멤버 알림 옵션 |
| CategoryMapping 일괄 재적용 (과거 거래 소급) | "거래 N건이 새 카테고리로 바뀌어요" 미리보기 + 확정 버튼 |
| 워크스페이스 탈퇴/삭제 | 2단계 확인 + 워크스페이스명 직접 입력 |
| AI 토글 끄기 | "이번 달 N건이 AI 파싱으로 들어왔어요. 끄면 다음부터 직접 입력해야 해요" 안내 |
| AI 동의 (첫 사용) | Value First 카드: "Gemini가 무엇을 하는지" 먼저 보여주고 동의 |
| API 키 생성 (`xef_*`) | scope 명시 + "이 키로 무엇을 할 수 있는지" 미리보기 |

원칙: **저위험 행동(기록·검토·카테고리 변경)은 매끄럽게, 고위험·비가역 행동은 일부러 무겁게.**

### 11.7 아키텍처 통합 — Rails 컴포넌트로 어떻게 도입할 것인가

- **partial vs ViewComponent**: 현재 레포는 ERB partial 기반. ViewComponent 도입은 별도 ADR로 결정. 본 제안은 우선 **partial로 표준화**한다.
- **토큰 위치**: `app/assets/stylesheets/application.tailwind.css`(또는 동등 진입점)에 `@theme` 블록으로 토큰 정의. Tailwind 4의 CSS-first config 활용.
- **다크 모드**: `prefers-color-scheme` + 사용자 토글 (`UserSetting#theme: auto|light|dark`). `light-dark()` CSS 함수 사용.
- **Hotwire**: 인라인 액션(카테고리 변경, 중복 표시, 용돈 마킹)은 모두 turbo-frame으로 부분 갱신. 풀 페이지 리로드 회피.
- **Stimulus**: chip toggle, slideover, sticky bar, segmented tabs를 각 1개 controller로.
- **i18n**: 11.5의 마이크로카피 표는 `config/locales/ko.yml`로 그대로 옮긴다. **하드코딩 금지**.
- **금액 헬퍼**: 모든 금액은 `format_amount(transaction.amount, kind: :spend|:income)` 헬퍼 경유. 컬러·부호·천단위 콤마·tabular-nums를 헬퍼가 책임.
- **모바일 퍼스트**: 디자인은 360~414px 폭 기준. 데스크탑은 max-width 640px로 중앙 정렬 (가계부의 본질이 모바일).

### 11.8 적용 로드맵 (Phased)

본 디자인 시스템을 한 번에 적용하지 않는다. 변경 영향이 크므로 단계로 나눈다.

| 단계 | 범위 | 산출물 |
|---|---|---|
| **Phase 0 — ADR 채택** | 본 11장을 별도 ADR로 승격 + 토큰·컴포넌트 카탈로그 동결 | `docs/decisions/ADR-XXX-design-system.md` |
| **Phase 1 — 토큰 도입** | `@theme` 블록 정의 + 다크/라이트 페어 + `format_amount` 헬퍼 + `ko.yml` 마이크로카피 | 시각 변화는 거의 없음. 토큰만 깔림 |
| **Phase 2 — 핵심 partial 표준화** | `screen_shell`, `context_header`, `hero_stat`, `transaction_row`, `empty_state`, `ai_badge_card`, `risk_notice` | dashboards / transactions / reviews 화면이 같은 컴포넌트로 표현 |
| **Phase 3 — 인라인 액션 & 검토 마찰** | TransactionRow 인라인 액션 / 검토 화면의 미해결 중복 차단 UX / RiskNotice 도입 | 사용자 행동 변화 첫 측정 |
| **Phase 4 — 빈 상태 & 라이브 컨텍스트** | 모든 EmptyState 통일, ContextHeader 인라인 컨텍스트(이번 달 델타 등) 도입 | 리텐션 지표 측정 |
| **Phase 5 — AI 채널 격리** | Gemini 출력물에 `AIBadgeCard` 일관 적용, AI confidence 노출, 동의 화면 Value First 재작성 | AI 신뢰·재사용 지표 |
| **Phase 6 — 회고/모니터링** | 디자인 시스템 적용 전후 지표 비교, 라이팅 사전 일관성 점검, 컴포넌트 거버넌스 회고 | 다음 사이클 인풋 |

### 11.9 거버넌스

- **컴포넌트 추가**: PR description에 "왜 기존 카탈로그로 못 푸는가" 1줄 필수 (P15).
- **마이크로카피 추가**: `ko.yml`에 키 추가 + 11.5 표 업데이트가 같은 PR에서 이뤄져야 함.
- **토큰 추가**: ADR 또는 supersede가 필요. 임의 추가 금지.
- **다크 모드 추가/변경**: light-dark 페어로만 추가. 한쪽만 정의 금지.
- **시각 모방 금지선**: 토스 동일 톤(브랜드 블루, 같은 라운드, 같은 플로팅 탭바 곡률, 같은 마이크로카피 어휘)을 그대로 가져오지 말 것. 모방으로 보일 경우 PR에서 거부.

### 11.10 측정 — 디자인 시스템이 잘 동작하는지 어떻게 알 것인가

| 지표 | 기대 방향 |
|---|---|
| 검토 화면에서 미해결 중복으로 인한 commit 실패 시도 | 안내 명확화로 감소 |
| pending → committed 전환에 걸리는 평균 시간 | 단축 |
| AI 카테고리 추천 채택율 | 상승 (AIBadgeCard + Easy to Answer로) |
| 빈 상태에서 다음 행동으로 이어진 비율 | 측정 시작 |
| 모바일 폭(< 414px)에서 가로 스크롤 발생 화면 수 | 0으로 수렴 |
| 마이크로카피 키 미정의 사용(하드코딩) | grep으로 모니터링, 0으로 수렴 |
| 카탈로그 외 partial 신설 빈도 | 거버넌스 통과 비율 추적 |

### 11.11 한 줄 결론

xef-scale을 위한 디자인 시스템의 핵심은 **"토스처럼 보이게"가 아니라 "사용자가 다음 행동을 추론하지 않게"** 다. 16개 정책(P0~P15)과 GPT 12원칙은 모두 이 한 줄의 *전술*이며, 윤리 보강은 매끄러움이 항상 옳지 않다는 *제동 장치*다.

xef-scale은 가계부이므로:

1. **숫자가 주인공** (이번 달 합계·예산·델타)
2. **한 화면 한 과업** (입력 / 검토 / 분류 / 회고)
3. **AI는 별도 채널** (보라 + ✨ + 신뢰도 표지)
4. **되돌리기 어려운 행동에는 의도적 마찰** (commit / 일괄 삭제 / 탈퇴)
5. **빈 상태가 다음 행동의 입구** (워크스페이스명 호명 + 1 CTA)
6. **마이크로카피는 토큰처럼 관리** (`ko.yml` 단일 출처)

이걸 위반하지 않는 한, 시각 표현은 자유다. 토스 스타일을 베끼지 말고, 토스의 *기율*만 가져간다.

## 참고

- 본 문서는 외부 앱 분석 노트이며 xef-scale 구현 결정과는 무관하다. 11장(디자인 시스템 제안)은 채택 시 `docs/decisions/`에 ADR로 승격되어야 비로소 권위가 된다.
- 7장(외부 자료 교차검증) 및 8~9장(GPT-5 병행 분석)의 인용은 모두 공개된 토스 공식 블로그/문서, 제3자 분석글, 외부 언론 보도에서 발췌·요약한 것으로, 원문에 대한 권리는 각 출처에 있다.
- 8~9장은 GPT-5 출력을 거의 원문 그대로 보존했으며 헤더 레벨과 각주 ID(`[g*]`/`[w*]`)만 본 문서 구조에 맞춰 조정했다.
- 11장은 토스 분석에서 *원리*만 추출해 xef-scale 도메인 어휘로 번역한 결과물이다. 토스 시각 표현(브랜드 컬러·라운드 곡률·플로팅 탭바·마이크로카피 어휘)은 의도적으로 사용하지 않는다.
