## Problem Statement

xef-scale Phase A는 가족 공유 UX, read-only API, 예산/반복결제를 구현했지만 아직 아무도 사용하지 않은 상태다. 현재 입력 방식(PC에서 Excel/PDF 명세서 다운로드 → 웹 업로드)은 모바일 사용자에게 마찰이 크다. 아내(핵심 타겟 사용자)가 실제로 사용하려면 모바일 퍼스트 입력이 필수이며, 수익화를 위한 유/무료 티어 구분이 필요하다.

## Demand Evidence

- 창업자 본인이 직접 사용하면서 "업로드가 병목"이라고 판단
- "모바일로 명세서를 쉽게 업로드할 방법이 마땅치 않다" — 직접 겪은 고통
- "금융앱에서 거래내역을 캡쳐한 걸 올리도록 하는 게 그나마 현실적" — 사용자 행동 기반 인사이트
- 편한가계부 2000만 다운로드 — SMS 파싱 기반 가계부의 시장 검증
- BYOAI 한계 인사이트: "구독 ≠ API 접근권", "일반 사용자에게 API 키 입력은 무리" — 다른 프로젝트에서 검증된 현실

## Status Quo

- 한국 가계부 시장: 마이데이터 연동(뱅크샐러드, 토스) vs SMS 파싱(편한가계부, 좋은가계부)
- 마이데이터: 규제 비용 높음, xef-scale에서 사용 불가
- SMS 파싱: 편한가계부가 200+ 금융 문자 양식을 정규식으로 커버 (10년 축적)
- iOS: SMS 자동 파싱 구조적으로 불가 → 복붙 또는 마이데이터만 가능
- 기존 xef-scale: 7개 기관별 텍스트 기반 파서 (Excel 중심, PDF는 미완성)

## Target User & Narrowest Wedge

**Target:** 창업자 본인과 아내. 여러 카드/은행을 쓰는 한국인 부부.

**Narrowest Wedge (Phase B):** 웹에서 금융 문자를 복붙하거나 스크린샷을 올리면, AI가 자동으로 파싱하여 거래 데이터로 저장. 기존 Excel 업로드와 병행.

## Constraints

- 마이데이터 API 규제 비용 → 사용 불가
- iOS SMS 자동 읽기 불가 → 복붙만 가능 (네이티브 앱 시점까지)
- BYOAI: 일반 사용자 대상 불가 (구독 ≠ API 접근권, API 키 진입장벽)
- SQLite 단일 DB (현재 규모에 충분)
- 기존 CI/CD: Docker + k8s (GitHub Actions)

## Premises

1. **모바일 퍼스트 입력이 핵심** — PC 명세서 다운로드는 파워유저 경로로 유지하되, 주 입력은 모바일(SMS, 스크린샷)이어야 한다

2. **무료 티어는 SMS 복붙 + 광고** — Excel/PDF 업로드 제거. 모바일 퍼스트. 안드로이드 자동 SMS 파싱은 네이티브 앱 트랙에서 추가

3. **자체 AI가 메인 유료 기능** — xef-scale이 Gemini Flash로 직접 파싱 (스크린샷, PDF, 영수증, SMS 텍스트). AI 비용은 구독료로 커버

4. **BYOAI는 파워유저용 escape hatch** — API 키 방식. 메인 포지셔닝이 아님. "앱 구독비 대신 내 API 키로" 수준

5. **유료 티어A(자체 AI)와 B(BYOAI) 모두**: 광고 제거 + 데이터 OS 심화 기능 (AI 분석, 트렌드, 절약 제안)

6. **새 입력 경로에는 기관별 파서를 만들지 않는다** — LLM이 범용 파싱. 편한가계부가 200개 정규식을 쌓는 동안, xef-scale은 LLM 하나로 "모든 양식" 커버

7. **Excel/PDF 파서 전체 제거, ParserRouter 제거** — 모바일 퍼스트. 입력은 SMS 복붙(텍스트)과 스크린샷/영수증(Vision)만. PDF는 모바일에서 마찰이 크고 스크린샷으로 대체 가능

8. **네이티브 앱은 Phase B가 아닌 별도 트랙** — SMS 자동 파싱, Share Extension 등 OS 통합이 필요할 때 시작. WWDC 2026 (6/8) 이후 Apple Extensions SDK 확인 후 결정

## Approaches Considered

### Approach A: "SMS First" — 네이티브 앱 + SMS 파싱 우선
- 안드로이드 네이티브 앱(Hotwire Native) 먼저, SMS 자동 파싱이 무료 핵심
- Effort: L (human: ~3주 / CC: ~3-4일)
- Risk: Medium — 네이티브 앱 + Play Store 배포가 새 영역
- Pros: 편한가계부와 동일한 입력 편의성, 무료로도 가치 제공
- Cons: iOS 여전히 복붙, 편한가계부와 SMS 정면 승부, 네이티브 유지보수

### Approach B: "AI Native" — Write API + AI 파싱 엔진 우선 ← SELECTED
- Transaction write API + 자체 AI 파싱 (Vision + text). 네이티브 앱은 이후
- Effort: M (human: ~2주 / CC: ~2-3일)
- Risk: Low — 기존 Rails + Gemini API 확장
- Pros: 네이티브 없이 웹에서 검증, 편한가계부와 다른 축(AI), iOS/Android 동시 지원
- Cons: SMS 자동 파싱 없음 (사용자 직접 입력), 모바일 UX가 웹앱 수준

### Approach C: "Hybrid" — Write API + 네이티브 동시
- 두 트랙 병렬. 무료=SMS, 유료=AI.
- Effort: XL (human: ~4주 / CC: ~5-6일)
- Risk: Medium-High — 스코프 큼, 검증 전 투자
- Pros: 가장 완전
- Cons: 아무도 안 써본 상태에서 과투자

## Recommended Approach

**Approach B: AI Native** — 아직 사용자 검증 전이므로 웹에서 바로 테스트할 수 있는 AI 파싱 엔진을 먼저 만든다. 아내 관찰 후 네이티브 앱 방향 결정.

### Phase B 태스크 (AI Native Input)

| # | 태스크 | 설명 | 의존성 |
|---|--------|------|--------|
| B0 | AI 파싱 벤치마크 | 실제 한국 금융 SMS 50건 + 스크린샷 20건으로 Gemini Flash 정확도 측정. Go/no-go: 텍스트 90%+, Vision 85%+. 목표: 텍스트 95%+, Vision 90%+. 미달 시 방향 재검토 | - |
| B1 | Transaction Write API | `POST /api/v1/transactions` — 기존 ApiKey 인증 + write scope. 기존 TransactionsController JSON 직렬화 재사용 | Phase A ApiKey, TransactionsController |
| B2 | AI 텍스트 파싱 엔진 | SMS/복붙 텍스트 → Gemini Flash → 구조화된 거래 데이터. 파싱 실패 시 사용자에게 원문 표시 + 수동 편집 폼 제공 | B0 통과 |
| B3 | AI Vision 파싱 엔진 | 스크린샷/영수증 이미지 → Gemini Flash Vision → 거래 데이터 | B0 통과 |
| B4 | 통합 파싱 UI | 웹에서 "텍스트 붙여넣기" or "이미지 업로드" → AI 파싱 → 기존 ParsingSession 리뷰 워크플로우(pending_review → commit) 재사용 → 저장. 파싱 결과 신뢰도 표시 + 수동 수정 가능. Gemini API 장애 시 "일시적으로 AI 파싱 불가" 안내 | B2, B3 |
| B5 | 기존 파서 전체 제거 | Excel/PDF 파서, ParserRouter, 기관별 파서 서비스 전체 제거 | B4 |

### Phase C 방향 (별도 설계 예정)

Phase B 검증 후 별도 /office-hours에서 상세 설계. 현재 방향:
- 구독 모델 (자체 AI / BYOAI 티어 + 광고 제거)
- AI 분석 엔드포인트 (트렌드, 절약 제안)
- API Rate Limiting + API 키 관리 UI
- MCP Write Tool (BYOAI 경로)
- 결제: Stripe 또는 토스페이먼츠 (한국 결제 지원 확인 필요)

```
│                    입력 경로                              │

│  ├── SMS 복붙 (텍스트) → 무료                             │
│  ├── 스크린샷 (이미지) → 유료A                            │
│  └── 영수증 사진 (이미지) → 유료A                         │
│                                                          │
│  🤖 BYOAI                                                │
│  └── POST /api/v1/transactions (직접 전송) → 유료B       │
└────────────┬─────────────────────────────────────────────┘
             │
             ▼
    ┌────────────────┐
    │ AI 파싱 엔진    │
    │                │
    │ 텍스트 → Gemini│
    │ 이미지 → Vision│
    └────────┬───────┘
             │
             ▼
    ┌──────────────────────────────────────────┐
    │         리뷰 워크플로우 (기존)             │
    │  ParsingSession → pending_review → commit │
    └──────────────────────┬───────────────────┘
                           │
                           ▼

```

### 수익 모델

| | 무료 | 유료A (자체 AI) | 유료B (BYOAI) |
|---|---|---|---|
| 입력 | SMS 복붙 (텍스트) | + AI Vision 파싱 (스크린샷, 영수증) | + API write (자기 AI로 전송) |
| 분석 | 기본 대시보드 | + AI 인사이트, 트렌드, 절약 제안 | + 동일 |
| 광고 | 있음 | 제거 | 제거 |
| 데이터 | 최근 3개월 | 무제한 | 무제한 |
| 가족 공유 | 1 워크스페이스 | 무제한 워크스페이스 | 무제한 워크스페이스 |

### LLM 파싱 vs 정규식 파싱 전략

편한가계부의 핵심 moat은 10년간 축적한 200+ 금융 문자 정규식 패턴. xef-scale은 이를 따라가지 않는다.

| | 편한가계부 (정규식) | xef-scale (LLM) |
|---|---|---|
| 새 카드사 추가 | 정규식 패턴 수동 추가 | 추가 작업 없음 |
| 양식 변경 대응 | 정규식 수정 필요 | 자동 적응 |
| 정확도 | 매칭 시 100%, 미매칭 시 0% | 95%+ 범용 |
| 비용 | 무료 | Gemini Flash: 1만 건 ~$0.04 |
| 유지보수 | 높음 (양식 변경마다) | 낮음 (프롬프트만 관리) |

## Privacy & Data Handling

- 금융 SMS/스크린샷에는 개인 금융 정보(거래 금액, 가맹점, 카드번호 일부)가 포함됨
- Gemini API로 전송 시 Google의 데이터 처리 정책 적용 — Gemini API는 유료 API 사용 시 학습에 사용하지 않음 (Google Cloud 약관)
- 원본 이미지는 파싱 완료 후 서버에서 삭제 (ParsingSession commit 시점)
- 사용자 동의: AI 파싱 첫 사용 시 "금융 데이터가 Google AI로 전송됩니다" 동의 UI 필요

## Success Criteria

- [ ] AI 텍스트 파싱: 한국 금융 SMS 10개 양식 이상에서 95%+ 정확도
- [ ] AI Vision 파싱: 금융앱 스크린샷 5개 기관에서 90%+ 정확도
- [ ] Write API: 외부에서 거래 데이터 생성 가능
- [ ] 아내가 웹에서 SMS 복붙 또는 스크린샷으로 거래 입력 가능 (관찰 검증)
- [ ] 기존 Excel/PDF 파서, ParserRouter 제거 완료

## What I noticed about how you think

- "모든 사용자가 AI를 구독하는 건 아니야" — BYOAI 포지셔닝의 구조적 한계를 다른 프로젝트에서 직접 겪고 정리해왔습니다. 이론이 아니라 실전에서 나온 인사이트입니다.
- "pdf 파서는 제거하자. 난이도가 너무 높더라" → ExpenseBot 연구 후 → "Vision AI로 교체" — 포기가 아니라 더 나은 방법을 찾는 사고 전환이 빠릅니다.
- "무료티어: 기존 buboo 앱처럼" — 경쟁 제품의 구체적 기능을 이미 체험하고 벤치마크로 쓰고 있습니다. 추상적 시장 분석이 아니라 사용자로서의 관찰입니다.
- "BYOAI 한계 요약"을 5개 항목으로 정리해서 가져온 것 — 인사이트를 구조화하는 습관이 있습니다. 이건 창업자로서 매우 강력한 특성입니다.
