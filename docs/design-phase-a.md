## Problem Statement

한국에서 여러 금융기관(카드/은행)을 사용하는 개인과 가족은 매달 지출을 추적하기 위해 각 금융앱에서 명세서를 다운로드한 후 엑셀에서 수동으로 정리해야 한다. 이 반복적이고 고통스러운 작업을 자동화하고, 가족 간 지출 현황을 공유할 수 있는 플랫폼이 필요하다.

## Demand Evidence

- 창업자(본인)가 직접 매달 엑셀에서 명세서를 수동 정리하는 고통을 겪고 있음
- 이 고통 해결을 위해 이미 운영 수준의 앱을 직접 개발함 (7개 기관별 파서, AI 분류, 멀티테넌트)
- "아내와 공유하기 위해서 만드는 거야" — 가족 공유가 핵심 동기
- 경쟁사 뱅크샐러드가 마이데이터 규제로 가족공유 서비스를 종료 — 시장 공백 존재

## Status Quo

- 각 금융기관 앱에서 명세서 다운로드
- 엑셀에 수동으로 복사/붙여넣기
- 카테고리 수동 분류
- 가족 간 공유는 파일을 직접 전달하거나 없음

## Target User & Narrowest Wedge

**Target:** 여러 카드/은행 사용하는 한국인 개인 + 가족/부부. 직접적인 첫 사용자는 창업자 본인과 아내.

**Narrowest Wedge:** 명세서 업로드 → 자동 파싱+분류 → 가족 워크스페이스에서 지출 현황 공유. 이것만으로 엑셀 수동 정리를 대체.

## Constraints

- 마이데이터 API는 규제 비용이 높아 사용 불가 → 수동 업로드 방식
- iOS에서 문자/푸시 내용 읽기 불가 (buboo의 한계)
- SQLite 단일 DB (Rails 8 기본) — 현재 규모(소수 동시 사용자)에는 충분. 동시 쓰기 병목 발생 시 PostgreSQL 마이그레이션은 Rails에서 직관적.
- 기존 배포 파이프라인: Docker + k8s (GitHub Actions CI/CD)

## Premises

1. 한국에서 여러 금융기관을 쓰는 개인/가족은 지출 추적에 고통을 느낀다
2. 수동 명세서 업로드가 마이데이터 API보다 가족 공유에 실용적이다 (규제 우회)
3. AI 자동 분류(Gemini) + 학습 루프가 수동 분류의 고통을 충분히 줄인다
4. 대상 사용자는 월 1회 명세서 다운로드+업로드를 감수할 의사가 있다
5. 웹앱(Rails + Hotwire)이 이 use case에 적합한 플랫폼이다

## 현재 기능 인벤토리 (이미 구현됨)

### 핵심 기능
- **파일 업로드 & 파싱**: ActiveStorage 기반 파일 업로드, SolidQueue 비동기 파싱
- **3단계 자동 분류**: CategoryMapping exact match → Category keyword match → Gemini AI 배치 분류
- **거래 관리**: 인라인 편집, soft delete/restore, CSV 내보내기, 일괄 작업
- **리뷰 워크플로우**: ParsingSession → pending_review → commit/rollback 상태 머신
- **중복 감지**: 날짜+금액+할부월 기반 DuplicateConfirmation

### 협업 기능
- **멀티 워크스페이스**: Workspace 기반 멀티테넌트, RBAC (owner/co_owner/member_write/member_read)
- **초대 시스템**: WorkspaceInvitation 토큰 기반, `join/:token` 경로
- **댓글**: 거래별 Comment 스레드
- **알림**: Notification 모델 (notification_type, read_at, action_url)
- **용돈 추적**: AllowanceTransaction 연결

### 분석
- **대시보드**: 월별/연별 지출 현황, 카테고리 비율
- **필터링**: 금융기관별, 카테고리별, 기간별

### 인프라
- **인증**: Devise + OmniAuth Google OAuth2
- **권한**: Pundit 정책
- **PWA**: manifest.json, service worker (기본 수준)
- **배포**: Docker + k8s, GitHub Actions CI/CD

## 가족 공유 데이터 모델 & 권한

기존 `WorkspaceMembership` 모델의 역할 기반 접근 제어:

| 역할 | 업로드 | 조회 | 카테고리 편집 | 멤버 초대 | 설정 변경 |
|------|--------|------|-------------|----------|----------|
| owner | ✓ | ✓ | ✓ | ✓ | ✓ |
| co_owner | ✓ | ✓ | ✓ | ✓ | ✓ |
| member_write | ✓ | ✓ | ✓ | ✗ | ✗ |
| member_read | ✗ | ✓ | ✗ | ✗ | ✗ |

가족 공유 시나리오: 부부 모두 `owner`/`co_owner`로 같은 워크스페이스에 참여. Pundit 정책으로 역할별 권한 제어 구현 완료.

## Gemini API 운영 고려사항

- **분류 방식**: 미분류 거래를 배치로 처리 (개별 호출 대비 비용 절감)
- **폴백 체인**: 5단계 모델 폴백 (상세: `gemini_category_service.rb` MODELS 상수 참조). gemini-3-flash-preview → gemini-2.5-flash-preview-09-2025 → gemini-2.5-flash → gemini-2.5-flash-lite-preview-09-2025 → gemini-2.5-flash-lite
- **학습 루프**: AI 분류 결과를 CategoryMapping에 저장 (source: gemini) → 동일 상점 재등장 시 API 호출 없이 1단계에서 즉시 매칭
- **API 장애 시**: 1단계(exact match) + 2단계(keyword match)로 가능한 만큼 분류. 미분류 거래는 "미분류" 상태로 저장 후 API 복구 시 재시도 가능
- **비용 추정**: 월 ~100건 거래 기준, 학습 후 API 호출은 신규 상점만 → 월 $0.01 미만

### Approach B: AI 에이전트 Write + 메신저 봇 (Phase 2)
Phase A에서 read-only REST API + MCP를 구축했으므로, Phase 2에서는 **write 기능 + 클라이언트 확장**:
- REST API + MCP write endpoint로 AI 에이전트가 명세서를 대신 업로드
- AI 스케줄링으로 "매달 1일 자동" 가능
- Phase A의 read API와 결합하면 완전한 "데이터 OS" API 완성
- **카카오톡/텔레그램 봇 (옵션)**: REST API 위의 경량 클라이언트. 파일 첨부 → 업로드(write API), "이번 달 얼마?" → 조회(read API), 예산 초과 push 알림. 아내가 매일 쓰는 앱 안에서 접근 → 웹앱 진입 마찰 제거. 카카오 개발자 등록 + 웹훅 서버 필요.

- Effort: S-M (API write만) ~ M (+ 봇 추가 시)
- Risk: Low-Med

### Approach C: BYOAI 전용 분석 (Phase 3) — 재정의됨

**기존 Approach C에서 stateful 기능(예산, 반복결제 감지)은 Phase A로 흡수됨.**

남은 것은 BYOAI에 위임할 분석 기능의 **지원 인프라**:
- MCP/API에 분석용 집계 endpoint 추가 (가족 멤버별 비교 데이터, 카테고리 트렌드)
- AI가 "지난 3개월 트렌드 분석해줘", "어디서 절약할 수 있어?" 등을 수행할 수 있도록 데이터 제공
- xef-scale은 **데이터를 정리하고 제공**하고, **판단과 제안은 사용자의 AI가 수행**

**원칙: xef-scale은 "정리하고 보여주는 것"까지. "판단하고 제안하는 것"은 BYOAI.**

| xef-scale이 하는 것 (stateful) | BYOAI에 맡기는 것 (분석) |
|---|---|
| 예산 설정 & 초과 알림 (Phase A) | "지난 3개월 트렌드 분석해줘" |
| 반복 결제 감지/목록화 (Phase A) | "이 구독들 중 안 쓰는 거 있어?" |
| 가족 멤버별 지출 집계 API (Phase C) | "우리 가족 누가 많이 쓰나?" |
| 월별/카테고리별 집계 API (Phase A) | 자유로운 자연어 질의 |

- Effort: S (집계 API endpoint 추가 위주)
- Risk: Low

## Strategic Positioning — "AI는 뇌, xef-scale은 기억" (Phase 1 아키텍처 원칙)

> **이 섹션은 철학적 비전이 아니라 Phase A의 아키텍처 결정을 직접 지배하는 원칙이다.**
> Task 6(MCP read), Task 7(예산), Task 8(반복결제)은 이 원칙의 직접적 구현이다.

**최대 경쟁자는 다른 가계부 앱이 아니라 ChatGPT/Gemini/Claude 자체.** 사용자가 명세서를 AI에 직접 던지면 1회성 분석은 즉시 가능. xef-scale이 AI 분석 기능을 내장해서 이길 수 없고, 이기려 해서도 안 된다.

**차별점: AI가 못하는 것 = 상태 유지(stateful).**
- AI 챗봇은 대화가 끝나면 데이터가 사라짐 — 12개월 누적 트렌드 추적 불가
- AI 챗봇은 가족 간 데이터 공유 불가 — 워크스페이스 개념 없음
- AI 챗봇은 분류 학습을 지속적으로 유지하지 못함 — CategoryMapping 누적 불가

**포지셔닝: xef-scale = 금융 데이터 OS.**
- 파싱, 누적 저장, 분류 학습, 가족 공유를 담당하는 데이터 레이어
- 분석은 사용자가 이미 구독 중인 AI(ChatGPT/Claude/Gemini)가 MCP/API로 수행
- 앱에 AI 비용을 전가하지 않음 → "또 구독?" 부담 없음

**기능 분류 원칙 — stateful vs stateless:**
- **xef-scale이 하는 것**: 정리, 저장, 보여주기, 알림 (stateful — AI로 대체 불가)
- **BYOAI에 맡기는 것**: 판단, 제안, 자연어 분석 (stateless — AI가 더 잘함)
- 모든 새 기능은 이 기준으로 분류: stateful이면 앱에, stateless면 API로 노출

**MCP API의 전략적 의미:**
- **Phase A (Read)**: AI가 누적 데이터를 조회하여 분석 (트렌드, 절약 제안, 예산 비교) — Day 1부터 "데이터 OS" 실현
- **Phase B (Write)**: AI가 명세서를 대신 업로드 (마찰 제거) — 완전한 BYOAI 루프
- 분석할 때마다 xef-scale을 거쳐야 함 → 자연스러운 lock-in

## Revenue Model

| 무료 | 유료 (구독) |
|------|------------|
| 업로드+파싱+분류 (기본) | 가족 공유 워크스페이스 |
| 단일 사용자 대시보드 | MCP/API 접근 (BYOAI — Bring Your Own AI) |
| 최근 3개월 데이터 조회 | 데이터 히스토리 무제한 보관 |

**구독 근거: AI 기능이 아니라 "정리된 데이터의 지속적 보관과 공유".** 이는 사용자의 기존 AI 구독과 겹치지 않는다.

## Recommended Approach

**A → B → C 순차 진행, 단 "데이터 OS" 기반은 Phase A에서 즉시 구축.**

- **Phase A (Task 1-8)**: 가족 공유 UX + read-only MCP + stateful 분석(예산/반복결제) — "데이터 OS" 포지셔닝을 철학이 아닌 코드로 실현
- **Phase B**: MCP write endpoint 추가 — AI가 명세서 업로드까지 대행, 완전한 BYOAI 루프 완성
- **Phase C**: 분석용 집계 API 확장 — BYOAI가 더 깊은 분석을 수행할 수 있도록 데이터 제공

**핵심 변경: Phase A에서 MCP read를 선제 구축함으로써, "데이터 OS"가 Phase 2 약속이 아니라 Day 1 현실이 됨.** 사용자 테스트에서 아내가 직접 업로드를 거부할 경우, Phase B를 앞당겨 AI 에이전트 업로드로 전환 가능 (read MCP가 이미 있으므로 write만 추가).
